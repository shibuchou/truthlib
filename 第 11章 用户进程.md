# 第 11章 用户进程

## 11.1 为什么要有任务状态段 TSS

一直以来我们的程序都在最高特权级0级下工作，这意味着任何程序都和操作系统平起平坐，可以改动任何系统资源。如果不改变这种现状的话，某个不听话的程序甚至可以给操作系统致命一击，后果将不可估量。

操作系统存在的目的之一就是资源管理，这里面就包括安全隔离的内容。要想服众，必须得拥有至高无上的权力，因此，被管理的程序不能具有太高的特权，必须要比操作系统的权力低，通常这个被管理的对象就是用户进程。我们把用户程序的特权级降级到3级，这样操作系统才能高枕无忧。

本章将开始介绍用户进程及相关的内容。

### 任务状态段TSS的起源

早在第5章中咱们已经简单介绍过TSS了，当时是为了介绍和I/O端口特权相关的I/O位图，此位图位于TSS中。现在接着和TSS有关的剩下部分。

### 11.1.1 多任务的起源
我们平时工作中所用的文档都是用某种应用程序创建的，应用程序一般是由某种编译器按照某种语言规则编译出来的，编译器也是程序，它为了编译出某操作系统平台上的应用程序，必须与宿命操作系统紧密耦合，因此编译器依赖宿主系统，“相当于”其应用程序，咱捋捋这个关系：文档是应用程序的应用，应用程序是编译器的应用，编译器又是操作系统的应用。那操作系统是谁的应用？当然是硬件的应用啦。

任何软件的运行都需要落实到硬件上，操作系统权力再高也只是软件，它的运行同样必须由硬件来支撑，只不过它和一般软件的区别是：操作系统是直接和硬件打交道的软件，普通的应用软件可没这个待遇。

操作系统能做什么，完全取决于硬件给它提供的功能支持。这就像驾驶汽车一样，如何驱动这辆车，要看车本身提供了哪些驱动的方法，比如有方向盘、油门等，而驾驶员也只是灵活应用这些驱动方法而已。我们的操作系统也只是众多计算机硬件的驾驶员。

### 11.1.2 LDT简介
按照内存分段的方式，内存中的程序映像自然被分成了代码段、数据段等资源，这些资源属于程序私有的部分，因此Intel建议，为每个程序单独赋予一个结构来存储其私有的资源，这个结构就是LDT。

LDT是Local Descriptor Table的缩写，即局部描述符表。LDT和GDT是对应的，GDT是全局描述符表，里面存放的是用于全局的内存描述符。描述符的功能就是描述一段内存区域的作用及属性，它只是对应内存区域的身份证。

LDT属于任务私有的结构，它是每个任务都有的，其位置自然就不固定。为了使用它，最起码得先能够找到它。我想您肯定想到了，GDT是全局唯一的结构，它的位置是固定且已知的，它已经由LGDT指令将其起始地址及偏移量存储到GDTR寄存器中。因此，LDT必须像其他描述符那样在GDT注册，之后便能够用选择子找到它。

描述符的作用是描述一段内存区域的属性，其中最重要的属性是内存区域的起始地址及偏移大小。描述符表是位于内存中的表格，因此，描述符表依然可以用描述符来表示。

LDT 是 **局部描述符表**，用于描述一个**任务私有的内存段**（如代码段、数据段等）。

与 GDT（全局描述符表）相对，GDT 是全局共享的，而 LDT 是每个任务独立的

LDT虽然是描述符“表”，为了在GDT中注册，必须也得为它找个描述符，用此描述符来描述某任务的LDT的起始地址及偏移大小，此描述符便称为LDT描述符，其结构如图11-1所示。

LDT描述符属于系统段描述符，因此S为0。在S为0的前提下，若TYPE的值为0010，这表示此描述符是LDT描述符。其他字段意义同段描述符相同，不再赘述。

CPU使用某个表，肯定不只是找到一个描述符就行了，描述符的目的是为了告诉CPU描述符所对应区域的起始地址及偏移大小。想想CPU是如何找到GDT的，在寄存器GDTR中的内容便是GDT的起始地址及偏移量大小，只要GDT被Lgdt指令加载后，CPU在GDTR中便能找到GDT。

LDT 的使用方式：LDT 本身也是一个段，需要在 GDT 中注册一个 **LDT 描述符**。使用 `lldt` 指令加载 LDT，操作数是 LDT 在 GDT 中的选择子。CPU 通过 **LDTR 寄存器** 来定位当前任务的 LDT。

![image-20251024094940015](F:\OS\xuexi\truth\图片\image-20251024094940015.png)

段选择子中有一个 **TI 位**（Table Indicator）：

- TI = 0：从 GDT 中选择描述符。
- TI = 1：从 LDT 中选择描述符。

这使得 CPU 能区分是从 GDT 还是 LDT 中找段描述符。

每个任务可以有自己的 LDT。任务切换时，操作系统需要：用 `lldt` 加载新任务的 LDT。更新 LDTR 寄存器。

LDT 最多支持 8192 个段描述符（因为选择子索引占 13 位）。如果每个任务都用 LDT，理论上最多支持 8192 个任务。

### 11.1.3 TSS （Task State Segment）的作用

TSS 是 Intel 为支持 **多任务切换** 而设计的一种 **硬件数据结构**。本质上是一块内存区域，用于保存 **任务的上下文（CPU 寄存器状态）**。每个任务（Task）理论上都应有一个 TSS。TSS 的结构由 CPU 规定，操作系统负责创建并初始化，CPU 负责在任务切换时自动读写。

#### **TSS 的作用**

**（1）保存任务上下文（寄存器快照）**

当任务被换下 CPU 时，CPU 会自动将当前寄存器状态保存到 TSS 中。当新任务被换上 CPU 时，CPU 会从新任务的 TSS 中恢复寄存器状态。包括：通用寄存器（eax, ebx, ...）、段寄存器（cs, ds, ...）、EIP、ESP、EFLAGS、CR3 等。

**（2）提供特权级栈（用于特权级切换）**

TSS 中保存了 **3 组栈指针（SS0~SS2 和 ESP0~ESP2）**，分别对应特权级 0、1、2。当从 **低特权级（如用户态）进入高特权级（如内核态）** 时，CPU 会自动从 TSS 中取出对应的栈指针（SS0:ESP0）作为新栈。这保证了特权级切换时，内核有自己的栈可用，防止用户态干扰内核态。

**（3）支持 CPU 原生任务切换（硬件任务切换）**

Intel 提供了硬件级任务切换机制：使用 `CALL/JMP + TSS 选择子` 或 **任务门** 进行任务切换。CPU 会自动保存旧任务状态到其 TSS，加载新任务状态从其 TSS。

但现代操作系统（如 Linux）**不使用这种机制**，因为效率低。

在 **CPU 原生任务切换模型** 中每个任务一个 TSS，任务切换就是 **TR 寄存器** 指向不同 TSS。，PU 自动保存/恢复寄存器。

但现代操作系统（如 Linux）**不使用这种模型**，而是所有任务共享一个 TSS（每 CPU 一个）。只使用 TSS 的 **ESP0 字段**（用于用户态→内核态栈切换）。任务切换由软件（内核）手动完成，效率高。

TSS 需要在 GDT 中注册一个 **TSS 描述符**。使用 `ltr` 指令将 TSS 选择子加载到 **TR 寄存器**。CPU 通过 TR 找到当前任务的 TSS。

I/O 位图用于控制用户态程序是否能访问特定 I/O 端口。实现 **端口级权限控制**。

上一个任务的 TSS 指针用于 **任务嵌套调用**（如任务 A 调用任务 B）。形成 **任务调用链**，支持 `iretd` 返回。

Busy 位（B 位）在 TSS 描述符中，表示任务是否正在运行。防止任务重入（自己调用自己）。

####  **现代操作系统对 TSS 的使用方式**

| 项目         | 原生 Intel 设计            | Linux 实际使用            |
| ------------ | -------------------------- | ------------------------- |
| TSS 数量     | 每任务一个                 | 每 CPU 一个               |
| 任务切换方式 | 硬件切换（CALL/JMP + TSS） | 软件切换（schedule 函数） |
| TSS 内容     | 全部寄存器                 | 只用 ESP0                 |
| TR 寄存器    | 每次切换任务都加载         | 启动时加载一次，永不更换  |
| 优点         | 简单，硬件自动完成         | 高效，灵活，可控          |

### 11.1.4 CPU 原生支持的任务切换方式

Intel 在硬件层面为“任务”这一概念量身定做的切换流程，核心思路非常直白：
“任务=一段内存+一个 TSS，换任务就是换 TR 指向的 TSS，CPU 包干所有寄存器现场搬运。”

具体步骤可以脑补成一条硬连线流水线：

1. 内核事先把新任务的 TSS 描述符填进 GDT，把当前旧任务的状态保存到它自己的 TSS 里。
2. 通过三种触发器——`CALL/JMP + TSS 选择子`、`CALL/JMP + 任务门`、`中断向量指向任务门`——把目标 TSS 选择子递交给 CPU。
3. CPU 微码开始跑：特权级检查、B 位检查、存在位检查，一通安检；把旧任务寄存器快照写回旧 TSS，B 位置 0；把新 TSS 里的整组寄存器搬到 CPU 内部，CR3、EIP、ESP、段寄存器一次到位；新 TSS 描述符 B 位置 1，TR 改成指向新 TSS；如果是 `CALL` 或中断引起的，还要把旧 TSS 选择子写进新 TSS 的“backlink”字段，并把新任务 EFLAGS.NT 置 1，形成一条“谁调我”的硬件链，好让 `IRET` 能沿着链原路返回。
4. 流水线结束，CS:EIP 已经是新任务的代码，CPU 继续取指，任务切换完成，全程没有操作系统一句软件保存/恢复汇编。

这套机制看起来让操作系统省了不少代码，但代价很大：

每条 `CALL/JMP` 都要在内存里来回搬运 104 字节以上的 TSS 数据，还要访问 GDT、写描述符 B 位，缓存几乎帮不上忙；任务多时 GDT 里 TSS 描述符随建随删，频繁重新加载 GDTR；指令本身内部微操作多达几百周期，远慢于纯软件把寄存器压到内核栈再换页表的做法；只支持“整任务”粒度，无法像软件那样在线程之间做精细调度。

因此 Linux 和本章项目都放弃这条硬连线，只在启动时喂给 CPU 一个“最小 TSS”满足特权级栈切换的硬性规定，之后调度器完全用软件完成上下文保存与恢复——把 TSS 当“跳板”而不是“现场仓库”。

### 11.1.5 现代操作系统采用的任务切换方式

Intel 已经把“任务切换”做成了硬指令，为什么 Linux 还要自己写一套调度器？答案可以浓缩成一句话——**“硬件太笨重，软件更轻更快更灵活”**。

首先看硬件方案的代价：每切一次任务，CPU 必须做一整套“安检＋搬运”——读 GDT、写 B 位、保存 104 字节 TSS、加载新 TSS、更新 NT 位、维护 backlink 链，全部串行，缓存帮不上忙，周期数以百计；任务一多，GDT 里还要动态增删 TSS 描述符，频繁重载 GDTR，开销继续放大。更致命的是，这条路径只能“整任务”硬切换，无法在线程之间做细粒度调度，也无法实现 Linux 里常见的任意混合调度策略（实时、CFS、cgroups 等）。

于是 Linux 干脆“阳奉阴违”：

1. 每 CPU 只建一个 TSS，启动时 `ltr` 一次就再也不换；
2. 只填三项内容：SS0、esp0 和 I/O 位图偏移，其余字段全部闲置；
3. 进程切换时，内核自己 `push/pop` 寄存器到内核栈，手动换页表，手动更新 `current`，比硬件路径少跑几十倍内存访问；
4. 用户态进内核态时，CPU 仍自动从同一 TSS 里取 esp0，满足特权级栈切换的硬需求——这是 Intel 的“底线”，无法绕过，但也就到此为止。

结果：任务切换从“几百周期 + 多次内存回写”变成“几十条指令 + 几次缓存命中”，速度提升一个数量级；同时软件可以随意玩线程、玩协程、玩任意调度类，不再受 8192 个描述符上限和硬件调用链的束缚。本章项目照搬 Linux 思路——TSS 只做“进入内核时拿栈”的跳板，调度器完全自己写，既合规又高效。

   `tss.ss0 = SELECTOR_K_STACK` —— 固定到内核数据段

   `tss.esp0 = 0` —— 先写 0，调度时动态改

   `tss.io_base = size` —— 偏移≥TSS 大小，表示“没有 I/O 位图”，用户进程全部端口被屏蔽

4. 进 GDT
   用 `make_gdt_desc()` 按 8 字节格式把 TSS 打包成描述符，直接写到 `0xc0000900 + 0x20`（第 4 项，紧跟显存段）；接着再把 DPL3 的用户代码段、数据段描述符写到 0x28、0x30 处，一次把后面用户进程要用的选择子也准备好。

5. 重新加载
   拼好 48 位操作数（limit=8×7-1，base 保留 0xc0000900），内联汇编 `lgdt` 让新表生效；`ltr %w0` 把 SELECTOR_TSS 装进 TR， Busy 位立即被 CPU 置 1，从此 TR 不再换。

6. 动态接口
   留下 `update_tss_esp(struct task_struct*)` 供调度器调用：只要把 PCB 页高地址（内核栈顶）写回 `tss.esp0`，下次该任务进中断时 CPU 就能拿到正确的内核栈。

做完这六步，TSS 的使命就结束——它永远挂在 TR 上，永远只负责“用户态→内核态那一跳”的栈切换；真正的寄存器快照、页表切换、线程调度全部交给软件调度器，后面写用户进程时再也不用碰 TSS 结构。

### 11.3.1 实现用户进程的原理

### 11.3.2 用户进程的虚拟地址空间

进程与内核线程最大的区别是进程有单独的 4GB 空间，这指的是虚拟地址，物理地址空间可未必有那么大，看似无限的虚拟地址经过分页机制之后，最终要落到有限的物理页中。我们需要单独为每个进程维护一个虚拟地址池，用此地址池来记录该进程的虚拟中，哪些已被分配，哪些可以分配。

与各个进程相关的数据，如果数据量不大的话，最好是存储在该进程的 PCB 中，这样便于管理。进程是基于线程实现的，因此它和线程一样使用相同的 pcb 结构，即 struct task_struct，我们要做的就是在此结构中增加一个成员，用它来跟踪用户空间虚拟地址的分配情况。具体新增的成员名是在 thread.h 中增加的，如代码 11-3 所示。

### 11.3.3 为进程创建页表和 3 特权级栈

进程与线程的区别是进程拥有独立的地址空间，不同的地址空间就是不同的页表，因此我们在创建进程的过程中需要为每个进程单独创建一个页表。我们这里所说的页表是“页目录表+页表”，页目录表用来存放页目录项 PDE，每个 PDE 又指向不同的页表。

页表虽然用于管理内存，但它本身也要用内存来存储，所以要为每个进程单独申请存储页目录项及页表项的虚拟内存页。

除此之外，之前创建的线程属于内核的线程，它们运行在特权级 0。和它们相比，用户进程还多了个特权级 3，大多数情况下，用户进程在特权级 3 下工作，因此，我们还要为用户进程创建在 3 特权级的栈。栈也是内存区域，所以，咱们还得为进程分配内存（虚拟内存）作为 3 级栈空间。

### 11.3.4 进入特权级 3

在**内核态（特权级 0）里，把用户进程“踢”进特权级 3**，并且让它第一次取指就落在自己的代码段、自己的栈、自己的页表上。核心手段是**伪造一次中断返回**——CPU 一执行 `iretd`，特权级就从 0 降到 3，路线图如下：

1. 唯一合法入口
   x86 不允许“平地”从高特权级跳到低特权级，**只有中断/异常返回**这一扇门。于是内核提前在用户栈（esp0）里布好一份 **struct intr_stack** 快照，假装“刚刚被中断过”。

2. 快照六件套
   `start_process()` 把 `intr_stack` 填成“即将返回的用户现场”：

   通用寄存器全 0（除 EIP/ESP）

   **CS = SELECTOR_U_CODE**（DPL=3，RPL=3）

   **SS = SELECTOR_U_DATA**（DPL=3，RPL=3）

   **ESP = 用户栈顶**（0xBFFF F000 + 4 KB）

   **EIP = 用户程序入口**（现用函数地址模拟）

   **EFLAGS = IF|MBS|IOPL_0**（开中断、允许被抢占、禁止 I/O）
   六样全部符合 iretd 的特权级检查公式：
   `CPL ≤ DPL 且 RPL ≤ DPL` → 0≤3 且 3≤3，放行。

3. 伪造返回过程
   把栈指针换成这份 **intr_stack** 地址 → 直接 `jmp intr_exit`（kernel.S）。
   `intr_exit` 按“中断退出”标准流程：

   连续 pop 恢复 GS～EAX

   `iretd` 最后一步弹出 **EIP/CS/EFLAGS** 时，发现 CS.RPL=3，硬件立即：
   – 把 CPL 改成 3
   – 从栈里再弹出 **ESP/SS**（已是用户栈）
   – 特权级切换完成，流水线继续取指，CS:EIP 指向用户代码。

4. 页表与栈已就位
   前面 `process_activate()` 已把 CR3 换成进程页目录，**intr_stack** 也映射在用户空间，iretd 不会触发缺页；用户第一条 push 就落在私有栈页。

5. 以后再进内核
   用户态只要中断/系统调用，CPU 自动从同一 TSS 取 **ESP0**，回到内核栈，现场保存路径与“真正被中断”完全一致，无需特殊处理。

### 11.3.5 用户进程创建的流程

用一张“流水线图”把用户进程从“源代码里的函数名”到“在 CPU 上跑起来”的全部里程碑串了起来。

   此时它仍是“内核线程”，只是入口函数是 start_process。
2. 页表与队列
   create_page_dir() 复制内核高 1 GB 映射，形成独立地址空间；
   PCB→pgdir 指向新页目录；
   list_append() 把任务挂到 thread_ready_list / thread_all_list，创建阶段结束。
3. 调度器接手
   时钟中断 → schedule() 从就绪链摘下该 PCB → process_activate()
   - 把 CR3 换成进程页目录（地址空间生效）
   - update_tss_esp() 把 PCB 顶端写进 TSS.esp0（中断栈准备完毕）
4. 跳板：switch_to → kernel_thread → start_process
   内核栈帧切换后 CPU 执行 kernel_thread，它立即调用 start_process(filename)。
5. 用户现场伪造
   start_process() 把 PCB 顶端的 intr_stack 填成“中断即将返回”的样子：
   CS/RPL=3，SS/RPL=3，EIP=filename 函数地址，ESP=用户栈顶，EFLAGS=开中断+IOPL=0。
   最后 `mov esp, intr_stack; jmp intr_exit` —— 栈指针指向伪造现场，iretd 一触即发。
6. iretd 落地
   iretd 发现 CS.RPL=3，硬件把 CPL 置 3，弹出用户栈，EIP 指向 filename；
   第一条用户指令开始执行，特权级 3 页表、栈、段描述符全部就绪。


**“内核线程→填私有页表→挂就绪队→调度器换 CR3→start_process 伪造 intr_stack→iretd 降到 3 级”**，用户进程由此正式跑起来。

任务：构造**第一份用户态上下文**，让 CPU 一 iretd 就能在 RPL=3 的环境下取指。

**2. 在 PCB 里定位 intr_stack**

线程创建时 intr_stack 被放在 PCB 最顶端；start_process 先把 self_kstack 回退 sizeof(thread_stack)，使 esp 指向 intr_stack 底部。

之后所有填充都用结构体偏移，保证 iretd 时硬件 pop 顺序与中断退出完全一致。

**3. 通用/段寄存器清零与初始化**

EAX~EDI、EBP、ESP_DUMMY 全部置 0——进程刚诞生没有历史。

GS 显式写 0：用户态不允许直接访问显存；即使忘写，iretd 特权检查也会把 GS 清 0。

DS/ES/FS 统一放 SELECTOR_U_DATA（DPL=3，RPL=3），保证数据段检查一次通过。

**4. 代码指针与栈指针硬塞进现场**

proc_stack->eip = function；

proc_stack->cs = SELECTOR_U_CODE（DPL=3，RPL=3）；

proc_stack->esp = **用户栈顶**——先调用 get_a_page(PF_USER, USER_STACK3_VADDR) 在私有地址空间申请一页，把“页末”当栈顶（向下增长）。
→ 此时页目录已切换，get_a_page 会把物理页映射到 0xBFFF F000~0xC000 0000，iretd 后第一条 push 就落在自己屋里。

**5. EFLAGS mask 位一次性算好**

EFLAGS_MBS (1<<1)  必须置 1（80286 遗留）；

EFLAGS_IF_1  (1<<9)  开中断，让用户进程能被时钟抢占；

EFLAGS_IOPL_0 (0<<12) 把 I/O 权限关死，用户态做不了 in/out。
组合值提前宏定义，直接 or 进 eflags 字段。

**6. 段选择子特权级一致性检查提前埋雷**

所有用户段描述符已在 11.2 放进 GDT，DPL=3；

选择子 RPL 硬编码为 3；

iretd 时硬件做 CPL≤DPL 且 RPL≤DPL 检查，条件满足→顺利降到 3 级，否则触发 #GP；
代码里把数值写死，确保不会拼错。

**7. 最后的“踢门”动作**

内联汇编 `movl %0, %%esp; jmp intr_exit`
– 把 esp 换成**伪造栈顶**；
– 跳去 kernel.S 的公共中断出口；
– intr_exit 连续 pop 恢复通用寄存器，最后 iretd 弹出 EIP/CS/EFLAGS + ESP/SS，硬件发现 CS.RPL=3，立即切换特权级、栈，用户进程第一条指令开始执行。

**8. 页表时机保证不出缺页**

上述全过程在 `process_activate()` 之后执行，CR3 已换成进程页目录；get_a_page 内部会建立 PTE，并置 PG_US_U | PG_RW_W | PG_P_1，iretd 前映射就已生效，因此**不会**在第一次 push 时触发 #PF。

**9. 与真正 ELF 加载的对应关系**

现在用函数地址模拟“入口点”，将来只需把
`proc_stack->eip = ELF header->e_entry;`
`proc_stack->esp = 用户栈顶 + 命令行大小;`
其余字段不变，同一套伪造现场就能直接 boot 任意 ELF。

### 11.3.7 bss 简介

**“bss 就是编译器给‘未初始化全局变量’留的座位——只占内存、不占文件，启动时由操作系统一次性清零，于是 C 代码里所有 `int g;` `static char buf[1024];` 都能放心当 0 用。”**

1. 文件视角
   .bss 段在 ELF 里没有实际内容，只有“多大”这个数；链接器把它合并进可加载的 DATA Segment，程序头里 `FileSiz` 小、`MemSiz` 大，差值就是 bss 大小。
2. 内存视角
   加载器根据 `MemSiz` 多扩一段，按页对齐后整页清零；此后这块内存与普通读写页无异，malloc 堆就紧挨着它往上长。
3. 内核视角
   本项目的用户进程暂时没用 ELF，但预留的“堆起始地址”仍放在 bss 之上——因为链接脚本里 _end 符号正好落在 bss 末尾，拿它当 `start_brk` 最省事；将来真做 ELF 加载时，把 `p_memsz - p_filesz` 那部分清零即可，流程与 Linux 完全一致。
