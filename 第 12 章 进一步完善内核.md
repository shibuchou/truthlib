# 第 12 章 进一步完善内核

## 12.1 Linux 系统调用浅析

用户进程权限被压到最低，想干任何事都得找内核；内核于是开一条**可控的通道**，让外面能安全地调用内部功能，这就是系统调用。

概念：只占用一个中断向量 0x80（int $0x80），避免把 IDT 撑爆。

eax 放“子功能号”，像拨号盘；ebx/ecx/edx/esi/edi 按序放参数。

内核根据 eax 查函数指针数组 syscall_table，跳转到具体 sys_xxx 实现。

两种接口

- glibc 的 syscall()：库函数，跨平台，参数个数任意。
- 老派 _syscall0~6 宏：直接嵌入汇编，废弃但简单，正好给我们当模板。

寄存器传参原因
若用栈，用户态→内核态要来回读两次内存；寄存器一步到位，更快也更省事。

## 12.2 系统调用的实现

### 12.2.1 系统调用实现框架

（1）用中断门实现系统调用，效仿 Linux 用 0x80 号中断作为系统调用

的入口。

（2）在 IDT 中安装 0x80 号中断对应的描述符，在该描述符中注册系统调

用对应的中断处理例程。

（3）建立系统调用子功能表 syscall_table，利用 eax 寄存器中的子功能号

在该表中索引相应的处理函数。

（4）用宏实现用户空间系统调用接口_syscall，最大支持 3 个参数的系统

调用，故只需要完成_syscall[0-3]。寄存器传递参数，eax 为子功能号，ebx

保存第 1 个参数，ecx 保存第 2 个参数，edx 保存第 3 个参数。

1. **0x80 中断描述符**（12.2.2）
   DPL=3，允许用户主动 int；入口指向 syscall_handler。

2. **syscall_handler 汇编例程**（12.2.4）
   统一压栈→取 eax 为索引→`call [syscall_table+eax*4]`→把返回值塞回栈→jmp intr_exit。

3. **syscall_table[]** 函数指针数组（12.2.5）
   下标=子功能号，元素=sys_xxx 地址；运行时注册，可动态扩展。

4. **_syscall0~3 用户宏**（12.2.3）
   内联汇编：eax=号，ebx/ecx/edx=参，int $0x80，返回 eax 值。

5. **首个 sys_getpid()**（12.2.5）
   仅一行 `return running_thread()->pid`，验证通路闭环。

   逻辑映射

| 表                     | 位置           | 作用               |
| :--------------------- | :------------- | :----------------- |
| IDT[0x80]              | interrupt.c    | 让用户能“拨号”     |
| intr_entry_table[0x80] | kernel.S       | 硬件跳转入口       |
| syscall_table          | syscall-init.c | 内核“派单”         |
| SYSCALL_NR 枚举        | syscall.h      | 用户可见的子功能号 |

以后做任何新系统调用都固定套路：
① 枚举加 `SYS_XXX` → ② 写 `sys_xxx()` 并注册进 syscall_table → ③ 用户侧用 `_syscall*` 包一层。
（12.2.6 / 12.2.7 以 getpid 为例完整走通）

参数摆放

1. **寄存器路线（主线）**
   eax=号，ebx=arg1，ecx=arg2，edx=arg3；`int $0x80` 后硬件自动压 ss/esp/eflags/cs/eip，handler 再压错误码+通用寄存器，形成统一的 intr_stack。
2. **栈路线（演示）**
   12.2.8 把参数压用户栈 → 内核从 int 栈帧里取 ss3/esp3 → 跨栈复制到内核栈；步骤繁琐，仅证明“内核态可安全读用户栈”。

12.2.7 的测试里，用户进程 `u_prog_a()` 只做两件事：
`prog_a_pid = getpid();`
内核线程帮它把结果打印出来——屏幕上出现 `prog_a_pid:0x2`，证明：

用户态 int 0x80 成功降权进入内核；eax=0 被正确解析为 SYS_GETPID；sys_getpid 返回当前 PCB 的 pid；返回值沿 eax→intr_stack→iretd 带回用户态；整个环路零报错，系统调用正式“出声”。

## 12.3 让用户进程“说话”

### 12.3.1 可变参数的原理

1. 调用者按 **右→左** 顺序把参数压栈，栈顶固定放 `format` 字符串。
2. `format` 里的 `%` 数量=**参数地图**；被调函数靠它知道后面有几块数据、每块多大。
3. 三个宏纯指针算术：
   - `va_start(ap,fmt)` → 让 ap 指向第一个可变参数
   - `va_arg(ap,type)` → 先 `ap+=sizeof(type)` 再返回旧地址，类型强转后解引用
   - `va_end(ap)` → 清指针防误用
4. 32 位下栈单元 4B，ap 每次 +=4 即可跳到下一参数；多传白传，少传踩雷——**仍是编译期行为**，只是看上去“可变”。

## 12.4 完善堆内存管理

### 12.4.1 malloc 底层原理

1. **arena**

   1 个或多个连续页框（4 KB 的整数倍）

   头部 12 B 元信息：desc 指针、large 标志、cnt 计数

   剩余空间按单一规格切成 N 个等长 mem_block

2. **mem_block**

   纯链表节点，只含 `list_elem free_elem`

   真实数据紧跟在节点后面，用户返回的是数据首地址

3. **mem_block_desc**

   全局数组 7 项，对应 16/32/64/128/256/512/1024 B

   维护 `free_list` 把所有同规格空闲块串在一起

   分配时从 free_list 弹块，释放时压回

**≤1024 B**：走 desc → arena → mem_block，**按规格就近适配**

**>1024 B**：large=true，整块页框直接返回，不再拆分

同一规格块用光时，再拿 1 页新建 arena，desc 指针指向同一描述符，形成“同规格 arena 集群”，free_list 长度动态增加。

1. size ≤1024？算 desc_idx → free_list 空就新建 arena → 拆块 → 弹块 → 返回数据地址
2. size >1024？页框数=DIV_ROUND_UP → malloc_page → 元信息标记 large=true → 返回 (arena+1)
3. 所有返回地址都跳过 12 B 元信息，用户拿到的是“纯净”数据区首地址。

**注意**：**不擦除页内数据**，只把位图相应位清 0，代表“空闲”。

#### 2. `page_table_pte_remove(uint32_t vaddr)`

**输入**：待取消映射的虚拟地址

**动作**：

1. `pte_ptr(vaddr)` 拿到 PTE 指针
2. `*pte &= ~PG_P_1` 只清 Present 位，其余属性保留
3. 内联汇编 `invlpg [vaddr]` 刷新 TLB 单条目

**理由**：

- 不必整页清零，CPU 见到 P=0 即认为无效
- 保留属性位，方便以后再次映射
- `invlpg` 比重载 CR3 轻量，且精准

#### 3. `vaddr_remove(enum pool_flags pf, void * _vaddr, uint32_t pg_cnt)`

**输入**：池类型、起始虚拟地址、连续页数

**动作**：

1. 选内核或用户虚拟内存池位图
2. 算起始位索引 → 循环 `pg_cnt` 次 `bitmap_set(..., 0)`

**结果**：位图清 0，代表“这些虚拟页已可回收”

malloc 时三步封装在 `malloc_page` 里：
`vaddr_get` → `palloc` → `page_table_add`

free 时同样封装成 `mfree_page`：
`pfree` → `page_table_pte_remove` → `vaddr_remove`

**顺序严格逆序**：
先还物理页（别的进程可立即再用）→ 再撕映射（CPU 立即感知无效）→ 最后清虚拟地址（防止再次分配冲突）。

## 总结

第十二章先用中断 0x80 搭起系统调用总线，实现用户↔内核的受控通道；接着把 write/printf/malloc/free 全部做成系统调用，用户进程第一次可以格式化输出和动态申请内存；最后引入一套基于 arena + 描述符的小块堆管理，让内核线程和用户进程各有独立内存池，达到 malloc/free 零泄漏、零碎片。
