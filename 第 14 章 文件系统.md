# 第 14 章 文件系统

## 14.1 文件系统概念简介

### 14.1.1  inode、间接块索引表、文件控制块 FCB 简介

#### 一、inode 是什么——“文件的身体坐标”

1. 定位职责
   每个文件/目录**必有一个** inode（i-node = index node），它保存：

   元数据：权限、属主、时间、文件大小

   数据块地址：12 个直接块 + 3 级间接块指针（一级、二级、三级）

2. 编号规则
   inode 在**整个分区**内统一编号 = 它在 inode_table[] 数组里的下标，0 号固定预留给根目录。

3. 与文件“一对一”
   “文件”在操作系统眼里就是 inode 编号；用户眼里的文件名只是“外号”，真正的实体是 inode。

#### 二、间接块索引表——“把文件做大”的机制 

为什么需要间接块，直接块只有 12 项，最大 12×4 KB = 48 KB。为了支持大文件，把“块地址”本身也放到数据块里，形成多级索引。

1. 三级索引结构（书本示例块大小 4 KB，地址 4 B，故 1 块可存 1 K 个地址）

   直接块：12 项 → 48 KB

   一级间接：第 13 项指向 1 个索引块 → 1 K×4 KB = 4 MB

   二级间接：第 14 项指向 1 个“索引块的索引块” → 1 K×1 K×4 KB = 4 GB

   三级间接：第 15 项 → 1 K×1 K×1 K×4 KB = 4 TB
   合计最大文件 ≈ 48 KB + 4 MB + 4 GB + 4 TB。

2. 地址计算流程（隐含重点）
   给定文件内偏移 → 先算落在哪一级范围 → 再按“索引块→索引块→数据块”顺序读盘，最多 4 次 I/O 就能拿到任意字节。

#### 三、FCB（文件控制块）

定义：凡用于“管理、控制、描述”文件的数据结构都叫 FCB；inode 就是 FCB 在类 UNIX 文件系统里的具体实现。

1. 包含字段

   身份：inode 编号

   安全：权限、属主

   时空：大小、时间戳

   地址：直接/间接块指针

   状态：打开计数、写禁止标志等（仅内存有效）

2. 与目录项的区别
   目录项 = {文件名, inode 号, 文件类型}，负责“把名字映射到 inode”；
   FCB/inode 负责“把 inode 映射到数据块”。二者配合才能实现“名→实”转换。

#### 四、三件套如何协作 

新建文件① 在 inode_bitmap 里分配一个空闲 inode 号 → ② 初始化 FCB（写大小、权限、直接块指针清 0）→ ③ 在父目录的数据块里新增一条目录项 → ④ 把 inode 写回磁盘。

1. 读写文件
   ① 用文件名搜目录项拿到 inode 号 → ② 读 inode 得到块地址列表 → ③ 根据偏移计算要访问的“第几个块”→ ④ 按直接/一级/二级/三级指针读出数据块 → ⑤ 完成用户缓冲区和磁盘块之间的拷贝。
2. 删除文件
   ① 在目录里把对应目录项清 0（或回收整个块）→ ② 把 inode 位图对应位清 0 → ③ 把数据块位图对应位清 0（块内容不用清零，下次分配直接覆盖）→ ④ 可选：把 inode_table 里该项清 0（调试目的）。

### 14.1.2 目录项与目录简介

#### 一、为什么 inode 里不存文件名

1. inode 只负责“**文件实体**”的元数据与块地址，是文件系统的“内部 ID”。

2. 文件名是给人看的，**可变、可重、可冲突**，若放 inode 里会导致：

   改名即改 inode，代价大；

   硬链接（多个名指向同一 inode）无法实现；

   目录下文件数受 inode 大小限制。

3. **文件名必须与 inode 解耦**，单独放在“目录文件”的数据区里——这就是目录项。

#### 二、目录项（dentry）

| 字段     | 长度（本实现） | 说明                                    |
| :------- | :------------- | :-------------------------------------- |
| filename | 16 B           | 用户可见的名字，以 '\0' 结束            |
| i_no     | 4 B            | 对应 inode 编号                         |
| f_type   | 4 B            | 文件类型：普通文件(1)、目录(2)、未知(0) |

大小固定 24 B，**不跨扇区**，便于线性扫描。

一个扇区 512 B 可放 21 个目录项，剩余 8 B 用 0 填充。

#### 三、目录的本质

1. **目录也是文件**，与普通文件共用同一套 inode 结构。

2. 区别仅在于 inode 指向的数据块里存放的是“**目录项数组**”，而不是用户数据。

3. 因此：

   目录有大小（i_size）= 所有目录项字节总和；

   目录可增长（写新文件 → 在数据块末尾追加目录项）；

   目录可收缩（删除文件 → 把对应目录项 f_type 置 0，形成“空洞”）。

#### 四、根目录的特殊性

每个分区**格式化时**就生成根目录 inode（编号固定 0），其数据块地址写死在超级块里。

根目录的“..”指向自己，形成“**目录树起点**”，破解“鸡生蛋”循环：

```
找文件 → 需父目录 → 父目录又需父目录 → … → 根目录（已知地址）
```

#### 五、从“文件名”到“文件数据”的 4 步流程

1. 路径解析：以 '/' 为起点，逐层拆分文件名。
2. 目录项查找：在当前目录的数据块里线性比对 filename。
3. 获得 inode 号：匹配成功后取出 i_no。
4. 读取 inode：用 i_no 定位 inode_table[]，拿到块地址，进而读写数据。

注意事项：

1. 目录项 ≈ “名片”（名字 + 房间号）

   inode ≈ “档案袋”（房间里的所有信息）
   一张名片可复印（硬链接），档案袋始终只有一份。

2. 广义 FCB 泛指一切管理结构；

   目录项是 FCB 的“**名字→inode**”部分，inode 是 FCB 的“**inode→数据**”部分。

3. **“.” 与 “..”**：

   每个目录**自动创建**的两个目录项，占用空间、参与大小计算；

   删除目录时若只剩这两项即视为“空目录”。

### 14.1.3 超级块与文件系统布局

一、超级块（Super Block）

1. 位置：固定写在每个分区的 **第 2 个扇区**（LBA=1），大小 1 扇区（512 B）。
2. 作用：挂载时 **第一扇区被读入内存**，据此就能算出所有后续元数据的位置。
3. 关键字段（共 13 项，52 B，剩余 460 B 填 0）

二、五大区域顺序
从低 LBA → 高 LBA 依次排列，**不可错位**：

1. 引导块（1 扇区）
   保留给 bootloader，文件系统不用，但**占位**。
2. 超级块（1 扇区）
   上面已讲。
3. 块位图（若干扇区）
   每 bit 对应数据区 1 个扇区，0=空闲，1=已用；大小 = 数据区扇区数 / 4096（向上取整）。
4. inode 位图（若干扇区）
   每 bit 对应 1 个 inode，0=空闲，1=已用；大小 = 最大文件数 / 4096。
5. inode 表（若干扇区）
   连续存放 `struct inode`，大小 = 最大文件数 × 88 B / 512（向上取整）。
6. 数据区（剩余所有扇区）
   起点 = `data_start_lba`，**根目录固定占用第 0 个数据块**，其余供普通文件/目录动态分配。

## 14.2 创建文件系统

### 14.2.1 创建超级块、i 结点、目录项

一、超级块结构体 `struct super_block`

1. 代码位置：`fs/super_block.h`

2. 设计原则：

   **必须 512 B**（1 扇区），不足用 `uint8_t pad[460]` 填 0。

   所有字段 `uint32_t`，共 13 项，便于磁盘对齐。

3. 关键字段：

| 字段             | 含义           | 典型值（80 MB 分区）     |
| :--------------- | :------------- | :----------------------- |
| magic            | 文件系统魔数   | 0x19590318               |
| sec_cnt          | 分区总扇区数   | 0xA521（42273 扇区）     |
| inode_cnt        | 最大文件数     | 4096                     |
| block_bitmap_lba | 块位图起始     | 0x41（紧跟超级块）       |
| inode_bitmap_lba | inode 位图起始 | 0x49                     |
| inode_table_lba  | inode 数组起始 | 0x4A                     |
| data_start_lba   | 数据区起始     | 0x2AA（根目录占第 0 块） |

------

二、inode 结构体 `struct inode`

1. 代码位置：`fs/inode.h`
2. 大小：88 B（< 512 B，**不跨扇区**即可）
3. 字段分工：

| 字段          | 宽度 | 作用                                  |
| :------------ | :--- | :------------------------------------ |
| i_no          | 4 B  | inode 编号（= 在 inode_table[] 下标） |
| i_size        | 4 B  | 文件：字节数；目录：目录项总字节      |
| i_open_cnts   | 4 B  | 内存引用计数，关闭文件时减 1          |
| write_deny    | 1 B  | 写锁，防多进程同时写                  |
| i_sectors[13] | 52 B | 0-11 直接块，12 一级间接块入口        |

三、目录项结构体 `struct dir_entry`

1. 代码位置：`fs/dir.h`
2. 大小：固定 24 B（**不跨扇区**，便于线性扫描）
3. 字段：

| 字段     | 宽度 | 说明                                        |
| :------- | :--- | :------------------------------------------ |
| filename | 16 B | 以 '\0' 结尾，不足补 0                      |
| i_no     | 4 B  | 对应 inode 编号                             |
| f_type   | 4 B  | 枚举：FT_UNKNOWN(0)/REGULAR(1)/DIRECTORY(2) |

四、三条落地规则

1. 地址对齐：所有结构体用 `__attribute__((packed))` 取消填充，**磁盘映像与内存映像一致**。

2. 扇区原子：超级块、位图、inode 表写入均以扇区为单位，**一次 512 B**，避免半扇区回写。

3. 预留策略：

   根目录 inode 编号固定 0，数据块固定占数据区第 0 块；

   位图最后一位若超出实际块数，**预置 1**（已占用），防止越界分配。

### 14.2.2 创建文件系统

一、整体流程（5 步 1 回滚）

1. 算大小
   按分区总扇区数，先算出

   inode 位图扇区数 = `ceil(4096 bit / 4096 bit per扇区)` = 1

   inode 表扇区数 = `ceil(4096 × 88 B / 512 B)` = 0x260

   块位图扇区数 = `ceil(剩余扇区数 / 4096)`（迭代一次，减掉自身占位）
   剩余扇区即为真正的数据区。

2. 写超级块
   把上一步结果填进 `struct super_block` 各字段 → `ide_write(part->start_lba + 1, &sb, 1)`
   （第 0 扇区留给引导块，超级块固定在第 1 扇区）

3. 填位图（内存→磁盘）

   块位图：第 0 位置 1（预留给根目录数据块），尾部多余 bit 全置 1（防越界）。

   inode 位图：第 0 位置 1（预留给根目录 inode）。
   写完立即 `ide_write()` 落盘，保证断电后仍能正确识别“已占用”。

4. 写 inode 表
   只初始化第 0 项（根目录）：

   `i_no = 0`

   `i_size = 2 × 24 B`（仅 “.” 和 “..” 两个目录项）

   `i_sectors[0] = data_start_lba`（数据区第 0 块）
   其余 4095 项全 0，表示空闲。

5. 写根目录内容
   在数据区第 0 块放 2 个目录项：

   “.” → i_no=0，类型=DIRECTORY

   “..” → i_no=0（根目录的父目录是自己）
   一次性 `ide_write()` 写入磁盘，格式化完成。

二、关键实现细节

单扇区原子写：位图、inode、目录项都先放 `buf[512]`，再一次性写盘，避免半扇区回滚。

尾部保护：块位图最后 1 字节多余位全置 1，防止将来分配越界块。

回滚策略：函数内用 `rollback_step` 记录失败点，出错时只回滚“已落盘”的资源（位图、inode），内存资源由调用者释放。

魔数校验：超级块 `magic = 0x19590318`，挂载时先读这 4 字节，不对就重新格式化。

### 14.2.3 挂载分区

一、流程总览（5 步）

1. 选分区
   通过 `list_traversal()` 遍历全局 `partition_list`，把名字（如 `"sdb1"`）与 `part->name` 匹配的分区指针赋给全局变量 `cur_part`——以后所有文件 API 默认操作它。

2. 读超级块
   `ide_read(cur_part->start_lba + 1, sb_buf, 1)`
   只读 1 扇区，验证 `magic == 0x19590318` 失败就报错，成功则把 13 个字段拷贝到内存 `cur_part->sb`（去掉 460 B 填充）。

3. 载位图（磁盘→内存）

   块位图：按 `sb->block_bitmap_sects` 申请内存 → 一次性 `ide_read` 整块。

   inode 位图：同理。
   两张位图从此常驻内核空间，**分配/回收只改内存**，回写时再落盘。

4. 建缓存
   `list_init(&cur_part->open_inodes)`
   初始化已打开 inode 链表，后续 `inode_open()` 先查链表里有无命中，没有才读盘，**避免重复 I/O**。

5. 打印 done
   `printk("mount %s done!\n", part->name)`
   挂载完成，后续 `sys_open/sys_read/...` 都通过 `cur_part` 访问元数据。

二、关键代码对象

| 内存对象                      | 来源         | 生命周期       | 作用                     |
| :---------------------------- | :----------- | :------------- | :----------------------- |
| `cur_part`                    | 全局指针     | 挂载后一直有效 | 默认操作分区             |
| `cur_part->sb`                | `sys_malloc` | 同上           | 内存超级块，快速访问     |
| `cur_part->block_bitmap.bits` | `sys_malloc` | 同上           | 内存块位图               |
| `cur_part->inode_bitmap.bits` | `sys_malloc` | 同上           | 内存 inode 位图          |
| `cur_part->open_inodes`       | 链表头       | 同上           | inode 缓存，加速二次打开 |

三、隐含规则

- 一张盘最多挂 1 个“当前分区”：再次 mount 会覆盖 `cur_part`。
- 位图 **整区读入**，后续 `bitmap_set/scan` 只在内存做，**回写由显式 `bitmap_sync` 完成**。
- 超级块 **只读一次**，以后改元数据（如 inode 分配）只改内存副本，**不回写超级块本身**（字段静态，无需回写）。
- 挂载失败（魔数错、内存不足）会释放已申请内存，**保持磁盘原样**。

## 14.3 文件描述符简介

### 14.3.1 文件描述符原理

一、4 个核心数据结构（内存中）

| 结构           | 所在            | 作用                           | 关键字段                                     |
| :------------- | :-------------- | :----------------------------- | :------------------------------------------- |
| PCB.fd_table[] | 每个进程私有    | 让不同进程有独立的 fd 名字空间 | int32_t，下标即用户可见 fd                   |
| file_table[]   | 全局唯一        | 记录“一次打开”的状态           | fd_pos 读写指针、fd_flag 标志、fd_inode 指针 |
| inode 队列     | 全局唯一        | 缓存常用 inode，减少读盘       | 链表，inode_tag 串联                         |
| inode 本身     | 磁盘 + 内存副本 | 描述文件实体                   | 大小、权限、块地址数组 i_sectors[]           |

二、3 步地址转换（fd → 数据）

1. fd → file_table 下标
   `uint32_t idx = current->fd_table[fd];`
   拿到进程私有的“打开实例”下标。
2. file_table[idx] → inode
   `struct inode *inode = file_table[idx].fd_inode;`
   得到文件实体描述符。
3. inode → 磁盘块
   用 inode.i_sectors[0..12] 直接/间接索引，算出 LBA，调 ide_read/ide_write。

三、2 个“为什么”

1. 为什么 fd 只是整数，而不是指针？

   隔离：进程间 fd 值可重复，互不影响。

   节省：PCB 只存 8 / 32 个整数，不必把庞大 file 结构搬进 PCB。

2. 为什么中间还要再拐一层 file_table？

   支持**多次打开同一文件**——每开一次就新建一个 file 结构，各自维护读写指针 fd_pos。

   支持**共享 inode**——多个 file 结构可指向同一 inode，实现硬链接与共享读写（需自己同步）。

一、inode 在磁盘上的坐标系

1. `struct inode_position`

   ```
   bool  two_sec;     // 是否跨扇区
   uint32_t sec_lba;  // 起始扇区号
   uint32_t off_size; // 扇区内字节偏移
   ```

2. `inode_locate(part, inode_no, &ipos)`
   输入 inode 编号 → 算出**绝对扇区+偏移**，供后续读/写使用。
   跨扇区判断：剩余空间 < 88 B 则 `two_sec = true`。

二、磁盘↔内存同步

`inode_sync(part, inode, io_buf)`

纯写盘：把内存 inode 复制到临时 `pure_inode`，**清掉 3 个仅内存字段**（`i_open_cnts/write_deny/inode_tag`）。

按 `ipos.two_sec` 决定读-改-写 1 还是 2 个扇区，**保证原子**。

三、内存缓存管理

`inode_open(part, inode_no)`

先到 `part->open_inodes` 链表查命中 → 命中则 `i_open_cnts++` 直接返回。

未命中：
– 临时把当前线程 `pgdir = NULL`，**强制从内核空间分配** inode 内存，实现“所有任务共享缓存”。
– 读盘（1/2 扇区）→ 拷入内存 → 插到链表**最前面**（利用局部性）。

返回**内存地址**，后续读写不再碰磁盘。

`inode_close(inode)`

`i_open_cnts--` 减到 0 时：
– 从链表摘除 → 同样 `pgdir = NULL` 保证用 **内核内存池释放** → `sys_free`。

不为 0 仅减计数，**真正的 inode 仍在缓存**。

四、新建 inode 快速模板

`inode_init(inode_no, new_inode)`

填 `i_no / i_size = 0 / i_open_cnts = 0 / write_deny = false`

把 `i_sectors[13]` 全清零，**不预分配任何数据块**（写时再分配）。

五、使用场景速查表

| 场景     | 调用链                      | 说明                      |
| :------- | :-------------------------- | :------------------------ |
| 创建文件 | `inode_sync()` 回写新 inode | 已分配 inode_no，需持久化 |
| 打开文件 | `inode_open()`              | 建立缓存+引用计数         |
| 关闭文件 | `inode_close()`             | 引用为 0 时释放缓存       |
| 写文件后 | `inode_sync()`              | 把更新的 size/块指针落盘  |
| 格式化   | `inode_init()`              | 批量生成空白 inode 模板   |

一、全局文件池

1. `get_free_slot_in_global(void)`
   - 扫描 `file_table[3 .. MAX_FILE_OPEN-1]`
   - 返回第一个 `fd_inode == NULL` 的下标（≥3），失败 -1
   - 保证**一个文件可被多次打开**，各自指针独立。

二、把池子下标装进 PCB

2. `pcb_fd_install(int32_t global_idx)`

在当前线程 `fd_table[]` 找最小值为 -1 的槽

写入 `global_idx` 并返回**用户可见 fd**（0~7）

失败 -1，表示进程表满。

三、拿 inode——inode 位图原子分配

3. `inode_bitmap_alloc(struct partition *part)`

`bitmap_scan(&part->inode_bitmap, 1)` 找 0 位

`bitmap_set(.., 1)` 立即置 1

返回位下标 = 新 inode 编号，失败 -1

**只改内存**，调用者视需要再 `inode_sync` 落盘。

四、数据块位图原子分配

4. `block_bitmap_alloc(struct partition *part)`

同上套路扫描 `part->block_bitmap`

返回**可用扇区 LBA**（而非位下标）：
`return part->sb->data_start_lba + bit_idx;`

同样内存级操作，**不立即写盘**。

五、同步位图——内存改 1 位→磁盘写 1 扇区

5. `bitmap_sync(struct partition *part, uint32_t bit_idx, enum bitmap_type btmp)`

计算该位落在哪个 512 B 块：`off_sec = bit_idx / 4096`

根据类型拿到内存地址 & 磁盘 LBA：

- `INODE_BITMAP` → `inode_bitmap_lba + off_sec`
- `BLOCK_BITMAP` → `block_bitmap_lba + off_sec`

一次性 `ide_write` 1 扇区，**保证断电后位图一致**。

六、使用场景速查

| 上层操作              | 内部调用链                                                  | 本层函数作用             |
| :-------------------- | :---------------------------------------------------------- | :----------------------- |
| `sys_open` 新建文件   | `inode_bitmap_alloc` → `block_bitmap_alloc` → `bitmap_sync` | 拿 inode 号 + 数据块 LBA |
| `file_write` 扩充文件 | `block_bitmap_alloc` → `bitmap_sync`                        | 动态拿新块               |
| `sys_unlink` 删除文件 | `bitmap_set(..,0)` → `bitmap_sync`                          | 回收块 & inode 位        |
| `sys_close`           | 无                                                          |                          |

#### 算法流程（5 步）

1. 根起步
   路径为 "/" 或 "/." 或 "/.." → 直接填 `parent_dir = &root_dir`，返回 0 号 inode。

2. 逐层拆
   循环调用 `path_parse()`：

   每拆一层 → 追加到 `record->searched_path`（带 '/'）

   更新 `pathname_depth` 与 `path_searched_depth` 用于后续比对

3. 线性搜
   对当前层 `name` 在当前目录数据块里**顺序比对** `dir_entry.filename`：

   命中且为目录 → `dir_close(old_parent)` → `dir_open(hit_inode)` 继续向下

   命中且为普通文件 → 结束搜索，返回 inode 号

   未命中 → 立即跳出，保留 `parent_dir` 不关闭（供创建用）

4. 断链停
   循环结束仍未找到 → 返回 -1，`record->file_type = FT_UNKNOWN`

5. 父目录留
   调用者必须 `dir_close(record->parent_dir)` 释放资源。

| 上层调用                   | 先调 search\_file    | 根据返回值处理             |
| -------------------------- | -------------------- | -------------------------- |
| `sys_open` 读文件          | 返回 ≥0              | 继续 `file_open(inode)`    |
| `sys_open` 带 O\_CREAT     | 返回 -1 且父目录存在 | 直接在 `parent_dir` 里创建 |
| `sys_mkdir`                | 同上                 | 同上，只是类型为目录       |
| `sys_unlink` / `sys_rmdir` | 返回 ≥0              | 确认类型后删除             |

#### 性能与边界

**顺序扫描**：目录项固定 24 B，每扇区 21 项，目前够用。

**最大深度**：仅受 `MAX_PATH_LEN` 512 B 限制。

**硬链接/重名**：由调用者再次验证，search_file 只负责“**存在+类型**”。

## 14.5 创建文件

### 14.5.1 实现 file_create

（1）文件需要 inode 来描述大小、位置等属性，所以创建文件就要创建其 inode。这就涉及到向inode_bitmap 申请位图来获得 inode 号，因此 inode_bitmap 会被更新，inode_table 数组中的某项也会由新的 inode 填充。

（2）inode->i_sectors 是文件具体存储的扇区地址，这需要向 block_bitmap 申请可用位来获得可用的块（在我们这里，为简化处理，1 块等于 1 扇区），因此 block_bitmap 会被更新，分区的数据区 data_start_lba以后的某个扇区会被分配。

（3）新增加的文件必然存在于某个目录，所以该目录的 inode->i_size 会增加个目录项的大小。此新增加的文件对应的目录项需要写入该目录的 inode->i_sectors[]中的某个扇区，原有扇区可能已满，所以有可能要申请新扇区来存储目录项。

（4）若其中某步操作失败，需要回滚之前已成功的操作。

（5）inode_bitmap、block_bitmap、新文件的 inode 及文件所在目录的 inode，这些位于内存中已经被改变的数据要同步到硬盘。

一、1 个核心函数——file_read
`int32_t file_read(struct file *file, void *buf, uint32_t count)`

1. 预检

   当前 `fd_pos + count > i_size` 自动截尾，**剩余 0 字节返 -1**（EOF 标志）

   正常：实际读出字节数（可能 < count）

1. `sys_read(int fd, void *buf, uint32_t count)`

**不分配块**：只读盘，不碰位图

一、1 个系统调用

```c
int32_t sys_lseek(int32_t fd, int32_t offset, uint8_t whence);
```

入口：文件描述符、偏移量、参照方式

出口：新的**文件内绝对偏移**（≥0）或 -1（越界）

二、3 种基准（whence）

| 宏       | 值   | 含义     | 计算式                    |
| :------- | :--- | :------- | :------------------------ |
| SEEK_SET | 1    | 文件开头 | new_pos = offset          |
| SEEK_CUR | 2    | 当前指针 | new_pos = fd_pos + offset |
| SEEK_END | 3    | 文件尾   | new_pos = i_size + offset |

三、2 条边界检查

`new_pos < 0` ⇒ -1

`new_pos > i_size - 1` ⇒ -1（**允许指向文件尾之后 1 字节**，即 EOF 位置，但**不允许越过**）
例：空文件 `i_size=0` 时，仅 `offset=0` 合法。

四、实现流程（4 句代码）

1. `idx = fd_local2global(fd)` 拿全局槽
2. 按 `whence` 三选一公式算 `new_pos`
3. 边界检查失败返 -1
4. `file->fd_pos = new_pos` 并返回新偏移

五、验证场景

| 操作                                | 结果                                |
| :---------------------------------- | :---------------------------------- |
| 写 24 B 后 `lseek(fd, 0, SEEK_SET)` | 指针回到头                          |
| 再 `read` 24 B                      | 重新读出全部内容，无需 close/reopen |

旧：读到 EOF 后想重读→**必须 close→再 open**

今：`lseek(fd, 0, SEEK_SET)` 一秒归零，**文件保持打开、缓存不失效**。

一、1 个系统调用

```c
int32_t sys_unlink(const char *pathname);
```

- 仅删**普通文件**，不能删目录
- 成功返 0，失败返 -1（文件不存在/被占用/是目录）

二、3 步回收链（由内到外）

1. 回收 inode 及所有数据块

   `inode_release(part, inode_no)`
   ‑ 收集 12 直接块 + 128 间接块 → 块位图全部清 0
   ‑ 释放一级间接表**本身**扇区
   ‑ inode 位图清 0（**不擦除 inode 内容**，下次覆盖即可）

2. 回收目录项

   `delete_dir_entry(part, parent_dir, inode_no, io_buf)`
   ‑ 在父目录里线性搜到对应项 → `f_type = FT_UNKNOWN`（清 0）
   ‑ 若该扇区只剩“.”和“..”**且不是根首块** → 整扇区回收（块位图清 0）

3. 关闭父目录

   `dir_close(record.parent_dir)` 释放内存

三、2 张位图同步规则

**每清 1 位立即 `bitmap_sync`**，只写改动的 512 B 扇区，断电安全

顺序：先数据块位图 → 后 inode 位图，**可重入失败**无影响（位 0 可重复清）

四、2 道保险门

1. 文件不存在或类型=目录 → 立即 -1
2. 文件仍在 `file_table[]` 被打开 → -1（**防止删正在写的文件**）

五、验证快照（xxd）

| 删除前 | 0x03（bit0,1=1） | 0x03           | 3 项（含 file1）  |
| 删除后 | 0x01（bit1=0）   | 0x01（bit1=0） | 只剩 2 项（. ..） |

二、5 步创建链（sys_mkdir 内部）

1. 搜路径
   `search_file()` 确保**同名不存在**且**中间目录全存在**
2. 拿资源
   - `inode_bitmap_alloc()` 得 inode_no
   - `block_bitmap_alloc()` 得 1 个数据块（存“.”和“..”）
3. 造 inode
   `inode_init()` → 类型目录，`i_size = 2×24 B`
4. 写目录内容
   数据块放 2 个固定目录项：
   - “.” → 自身 inode_no
   - “..” → 父目录 inode_no
     一次性 `ide_write` 落盘
5. 挂入父目录
   `create_dir_entry()` + `sync_dir_entry()` 把新目录名写进父目录
   → 父目录 `i_size += 24 B` → `inode_sync` 回刷
   → 位图同步（inode + 块）

三、4 步删除链（sys_rmdir 内部）

1. 判存在与类型
   `search_file()` 必须返回目录且 inode 存在
2. 判空目录
   `dir_is_empty()`：仅含 "." 和 ".." → `i_size == 2×24 B`
3. 删目录项
   `delete_dir_entry()` 把父目录里对应项 `f_type` 清 0
   → 若该扇区只剩 "." ".." 且非根首块 → **整扇区回收**
4. 回收 inode 与块
   `inode_release()`：清 inode 位图 + 释放数据块位图
   → 关闭目录内存对象

一、3 个用户层 API（原型对标 POSIX）

| 函数     | 原型                                | 作用                     |
| :------- | :---------------------------------- | :----------------------- |
| opendir  | `DIR *opendir(const char *name)`    | 打开目录流，返目录指针   |
| readdir  | `struct dirent *readdir(DIR *dirp)` | 读下一项，返 dirent 指针 |
| closedir | `int closedir(DIR *dirp)`           | 关闭并释放资源           |

本实现用 `struct dir` 代替 `DIR`，`struct dir_entry` 代替 `struct dirent`。

二、3 个内核实现（sys_ 前缀）

1. `sys_opendir(path)`

   根目录特殊路径（"/" "/." "/.."）直接返 `&root_dir`

   其余调 `search_file` → 必须存在且类型 = DIRECTORY

   命中后 `dir_open(inode_no)` → 返回 `struct dir *`（含 inode 指针与游标 `dir_pos=0`）

2. `sys_readdir(dir)`

   单纯封装 `dir_read(dir)`：
   ‑ 遍历目录所有块（12 直接 + 128 间接）
   ‑ 用 `dir_pos` 做游标，跳过已返回项（`f_type==0` 也跳过）
   ‑ 找到合法项后更新 `dir_pos += 24` 并返回其指针

   到尾返 `NULL`

3. `sys_closedir(dir)`

   根目录指针直接返 0

   普通目录 `dir_close(dir)` → `inode_close` + `kfree(dir)`

三、2 个验证场景（main 演示）

1. 遍历 `/dir1/subdir1`

   ```
   directory .
   directory ..
   regular   file2
   ```

   证明能区分文件与目录，且跳过空槽。

2. `sys_rewinddir(dir)`
   把 `dir_pos` 清零即可重新从头读取，**免 close/open**。

四、关键实现细节

目录项 **不跨扇区**，`dir_read` 每次只读 512 B

游标 `dir_pos` 以**字节**为单位，步长 24 B
