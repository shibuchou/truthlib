# 第 15 章 系统交互

## 15.1 fork 的原理与实现

###  15.1.1 什么是 fork

1. 作用：把「一个进程」复制成「两个几乎一模一样」的进程——父进程与子进程。
2. 返回值“一次调用，两次返回”：
   - 在父进程里返回「子进程 pid（>0）」
   - 在子进程里返回「0」
   - 失败返回「-1」
3. 时空观：fork 之后两条执行流从同一断点向下并行；代码段、数据段、堆、栈、打开文件等全部克隆，但各用各的地址空间（写时复制）

- 用经典示例演示“一个 printf 前后出现两次”的现象，破除“fork 必定走 if-else 不同分支”的误解。
- 给出 fork 的“叉子”模型：一把柄在 fork 点分叉成两条相同柄（进程），强调“进程是运行中的程序”。sys_fork

原理

1. 复制：把父进程 PCB、内核栈、页表、用户空间、文件打开表等全部克隆到子进程。
2. 微调：给子进程发新 pid、清空运行时间、状态置 READY、parent_pid 指向父进程。
3. 伪造现场：把子进程的中断栈 eax 置 0，保证它从 intr_exit 返回时拿到返回值 0。
4. 就绪：把子进程塞进就绪队列，调度器以后选它运行。

步骤（自上而下）
① 分配 1 页内核空间作为子进程 PCB
② copy_process()
├─ a. memcpy 整页：复制父 PCB＋内核栈
├─ b. 分配新 pid，改关键字段（pid/ticks/status/parent_pid）
├─ c. 复制用户虚存位图 → 子进程独享
├─ d. create_page_dir() → 给子进程建页表（内核部分共享，用户部分待拷）
├─ e. copy_body_stack3() → 按父进程用户位图逐页拷数据（内核缓冲中转）
├─ f. build_child_stack() → 构造 thread_stack，把返回地址改成 intr_exit，eax 置 0
└─ g. update_inode_open_cnts() → 所有打开文件的 i_open_cnts++
③ list_append 到就绪队列 & 全局队列
④ 返回「子进程 pid」给父进程

### 15.1.2 fork 的实现

一、总览
内核入口：sys_fork() → 复制父进程 → 把子进程扔进就绪队列 → 返回子进程 pid。
整个过程分「5 步 1 链」：5 个独立函数顺序完成，1 条失败回滚链（本小节未展开回滚，代码里用 ASSERT/return -1 保证安全）。

二、关键函数与原理

| 函数                          | 原理一句话                                                   | 本节课新增重点                                        |
| :---------------------------- | :----------------------------------------------------------- | :---------------------------------------------------- |
| copy_pcb_vaddrbitmap_stack0() | 把父 PCB、内核栈、用户虚存位图整页克隆，再「微改」成子进程身份 | 虚存位图必须重新申请一页，不能让父子共用              |
| create_page_dir()             | 为子进程新建页目录：内核 1 GB 共享、用户 3 GB 留空待拷       | 沿用第 10 章实现，无需再写                            |
| copy_body_stack3()            | 按「父用户位图」逐页搬运已用物理页 → 子用户空间              | 用内核页做中转，页表来回切换，保证 pte 落到子进程页表 |
| build_child_stack()           | 伪造中断返回现场：eax=0、eip=intr_exit，让子进程「从中断返回」 | 把 thread_stack 链到 intr_exit，eax 置 0 即返回值     |
| update_inode_open_cnts()      | 把父子进程共同打开的所有文件引用计数 +1                      | 防止 close 时提前释放 inode                           |

三、步骤拆解（自上而下看代码顺序）

1. 父进程陷入内核 → sys_fork()
2. 申请 1 页内核空间做子进程 PCB
3. copy_process() 一条龙
   3.1 copy_pcb_vaddrbitmap_stack0()
   \- memcpy 整页（PCB+内核栈）
   \- fork_pid() 发新 pid
   \- 虚存位图重新申请并拷贝比特
   3.2 create_page_dir() → 得子页目录（内核共享）
   3.3 copy_body_stack3()
   \- 遍历父用户位图，每找到一页：
   ‑ 父页内容 → 内核临时页
   ‑ 切到子页表 → get_a_page_without_opvaddrbitmap()
   ‑ 内核临时页 → 子用户页
   ‑ 切回父页表
   3.4 build_child_stack()
   \- 子 PCB 顶部找到 intr_stack
   \- intr_stack.eax = 0          // 子进程返回值
   \- 构造 thread_stack 指向 intr_exit
   3.5 update_inode_open_cnts()
   \- 对所有 fd≥3 的打开文件，全局 file_table[].i_open_cnts++
4. 把子进程 general_tag/all_list_tag 插到就绪队列 & 全局队列
5. 返回子进程 pid 给父进程；子进程以后被调度时从 intr_exit 弹出，用户态拿到返回值 0

### 15.1.3 添加 fork 系统调用与实现 init 进程

一、本节任务清单

1. 把 `sys_fork` 注册成系统调用，用户态可用 `fork()`。
2. 建立第一个「用户态」进程——init，抢占 pid=1，以后所有用户进程的“祖宗”。
3. init 一启动就 `fork()`，父进程负责“收尸”，子进程跑 shell，形成“init→shell→外命令”链条。

二、fork 系统调用 3 步曲（回顾式速办）
① syscall.h 加枚举 `SYS_FORK`
② syscall.c 包壳 `pid_t fork(void){ return _syscall0(SYS_FORK); }`
③ syscall-init.c 把 `syscall_table[SYS_FORK]` 指向 `sys_fork`
→ 用户代码 `int pid = fork();` 即可用。

三、init 进程设计要点

- 身份：必须是「用户进程」而非内核线程，所以用 `process_execute(init, "init")` 创建。
- 时机：在 `thread_init()` 里最先生成，确保 pid=1 被它独占。
- 逻辑：
  ├─ 父进程（init 本身）：无限 `wait(&status)`，给任何孤儿/僵尸收尸，打印回收信息。
  └─ 子进程：调用 `my_shell()`，成为控制台交互入口。

四、代码落地速览
main.c

```c
void init(void) {
    uint32_t ret_pid = fork();
    if(ret_pid) {                // 父进程——init 守护
        int status;
        while(1) {
            int child_pid = wait(&status);
            printf("init recieve child %d status %d\n", child_pid, status);
        }
    } else {                     // 子进程——控制台
        my_shell();
    }
}
```

thread.c

```c
void thread_init(void) {
    ...
    process_execute(init, "init"); // ① 抢 1 号 pid
    make_main_thread();            // ② 主线程变内核线程
    idle_thread = ...              // ③ 空闲线程
}
```

五、运行 snapshot
启动信息中最早出现

```
init recieve child 4 status 0
```

表明 shell（pid=4）已退出并被 init 回收；此后任何用户进程若父进程先死，都会过继到 init，由上述 `wait` 循环统一“火化”。

## 15.2 添加 read 系统调用，获取键盘输入

一、目标
让 `read(fd, buf, count)` 既能读文件，也能从**键盘**（stdin）拿字符，完成“人机交互”最后一公里。

二、关键改动点
只动 **sys_read** 一个函数，内部加三路分流：

1. `fd == stdin_no` → 走键盘缓冲区 `kbd_buf`（环形队列 `ioqueue`）
2. `fd == 管道读端` → 走 `pipe_read()`（15.7 预留）
3. 其它 → 原 `file_read()` 不变

三、代码变更（fs/fs.c）

```c
int32_t sys_read(int32_t fd, void *buf, uint32_t count){
    if(fd == stdin_no){                 // 新增分支
        char *p = buf;
        uint32_t cnt = 0;
        while(cnt < count)
            p[cnt++] = ioq_getchar(&kbd_buf);
        return cnt ? cnt : -1;
    }
    /* 管道分支稍后补 */
    return file_read(...);              // 原文件逻辑
}
```

四、用户侧 3 步曲（回顾）
① `syscall.h` 加 `SYS_READ`
② `syscall.c` 包壳 `read()` → `_syscall3(SYS_READ, fd, buf, count)`
③ `syscall-init.c` 注册 `syscall_table[SYS_READ] = sys_read`

五、瞬时验证
shell 里首次能用

```c
char c;
read(STDIN_FILENO, &c, 1);
```

直接从键盘拿键值，回显可控，为下一步 **readline / shell 编辑** 打好地基。

## 15.3 添加 putchar、clear 系统调用

一、目标
给用户态提供“最小 IO 工具箱”：

- `putchar(c)` 即时输出一个字符
- `clear()` 一键清屏并把光标归零（模拟 Ctrl-L）

二、实现策略

1. 内核层已有现成函数
   - `console_put_char()` → 直接复用
   - `cls_screen()`（print.S 里汇编实现）→ 清屏 + 置光标
2. 不再新增 sys_xxx，直接让系统调用表指向上述函数，零封装成本。

三、3 步注册
① `syscall.h` 枚举追加

```c
SYS_PUTCHAR, SYS_CLEAR
```

② `syscall.c` 用户桩

```c
void putchar(char c){ _syscall1(SYS_PUTCHAR, c); }
void clear(void){ _syscall0(SYS_CLEAR); }
```

③ `syscall-init.c` 绑定

```c
syscall_table[SYS_PUTCHAR] = console_put_char;
syscall_table[SYS_CLEAR]   = cls_screen;
```

四、效果

- 用户代码可链式 `putchar('A'); putchar('\n');`
- shell 快捷键 Ctrl-L 直接 `clear();` 清屏并重绘提示符，为后续行编辑、菜单打印打底。

## 15.4 实现一个简单的 shell

### 15.4.1 shell 雏形

一、目标
让 init 的子进程化身“可交互外壳”：

- 打印提示符 `[rabbit@localhost /]$ `
- 读入一行命令（支持退格、回车）
- 回显并解析，为后续内/外部命令铺路。

二、核心数据结构与宏

```c
#define cmd_len     128          // 一行最长
#define MAX_ARG_NR  16           // 含命令名最多 15 参数
static char cmd_line[cmd_len];   // 原始输入缓冲区
static char cwd_cache[64] = "/"; // 当前目录缓存（给提示符用）
```

三、关键函数

1. `print_prompt()`
   输出 `[rabbit@localhost 当前目录]$ `

2. `readline(char *buf, int32_t count)`

   - 循环 `read(stdin_no, &ch, 1)` 单字符读
   - 处理 `\b` 退格（不回删过提示符）、`\r`/`\n` 结束并回车换行
   - 可见字符即时 `putchar` 回显
   - 末尾自动补 `\0`

3. `my_shell()` 主循环

   ```
   while(1){
       print_prompt();
       memset(cmd_line, 0, cmd_len);
       readline(cmd_line, cmd_len);
       if (cmd_line[0] == 0) continue;   // 纯回车

   }
   ```

   

四、启动链

- `thread_init()` → `process_execute(init, "init")`
- init 里 `fork()` 后子进程调 `my_shell()`，父进程死等收尸。
- 主线程清屏并打印首提示符，正式交出控制台。

### 15.4.2 添加 Ctrl+u 和 Ctrl+l 快捷键

一、目标
在 `readline()` 里把两个 Linux 经典快捷键做进内核：

- **Ctrl-L**：清屏 → 重绘提示符 → 恢复已输入内容（不丢命令）
- **Ctrl-U**：把**当前行已输入内容全部清空**（光标回到行首）

二、实现策略

1. 不改动键盘中断底层，**在 `readline` 的字符循环里识别控制字符**即可；减少中断负担。
2. 控制字符生成方式：
   键盘驱动把 *Ctrl+字母* 转成 `ascii = 字母 - 'a'`（0–31 控制区），
   因此 `Ctrl-L` → `0x0C` (`'l'-'a'`)，`Ctrl-U` → `0x15` (`'u'-'a'`)。
3. 收到后**就地处理**，不回写缓冲区，立即刷新屏幕。

三、代码改动（shell.c `readline` 内部）



```c
switch (*pos) {
case 'l' - 'a':                    // Ctrl-L
    *pos = 0;                      // 丢弃该字符
    clear();                       // 1. 清屏
    print_prompt();                // 2. 提示符
    printf("%s", buf);             // 3. 重打已输入
    break;

case 'u' - 'a':                    // Ctrl-U
    while (pos > buf) {            // 回退到行首
        putchar('\b');             // 光标左移
        *(--pos) = 0;              // 缓冲区回删
    }
    break;
...
}
```

四、效果

- 随时按 **Ctrl-L** 屏幕瞬间干净，提示符+已敲命令原地恢复。
- 按 **Ctrl-U** 当前行被“剃头”，光标回到行首，可重新输入。

### 15.4.3 解析键入的字符

一、目标
把 `readline` 得到的**原始字符串**拆成
`argv[] = {"命令", "arg1", "arg2", ...}`
供后续 `exec / 内建命令` 使用。

二、核心数据结构

```c
#define MAX_ARG_NR 16
char *argv[MAX_ARG_NR];   // 全局，供 exec 直接继承
int32_t argc;
```

三、关键函数 `cmd_parse()`
输入：原始缓冲区 `cmd_str`、分隔符 `token`（空格）
输出：填充 `argv[]`，返回参数个数 `argc`

四、算法步骤（就地解析，无额外内存）

1. 先清 `argv[]` 全 NULL
2. 双指针扫描
   - 跳过连续分隔符
   - 遇非分隔符 → 记录 `argv[argc]` 起始地址
   - 继续到下一个分隔符或结尾 → 把分隔符改 `\0`（截断单词）
   - `argc++`
3. 超上限返回 -1，上层打印错误

五、代码骨架

```c
static int32_t cmd_parse(char *cmd_str, char **argv, char token) {
    int argc = 0;
    for (int i = 0; i < MAX_ARG_NR; i++) argv[i] = NULL;

    char *next = cmd_str;
    while (*next) {
        while (*next == token) *next++ = 0;   // 跳过空格
        if (*next == 0) break;
        if (argc >= MAX_ARG_NR) return -1;
        argv[argc++] = next;                // 记录单词首地址
        while (*next && *next != token) next++; // 找单词尾
    }
    return argc;
}
```

输入：

```

```

解析后：

```
argv = {"ls", "-l", "dir1"}; argc = 3;
```

shell 主循环打印测试可见，单词被正确分割。

### 15.4.4 添加系统调用

一、本节任务
把之前已经写好的 **sys_xxx** 内核函数全部注册成“用户态能直接打电话”的系统调用，为 shell 提供
ls / cd / pwd / ps / mkdir / rmdir / rm / open / close / lseek / getcwd / opendir / closedir / readdir / rewinddir / stat / clear / putchar 全套 API。

二、套路 3 步曲（一次性批量完成）

1. 枚举加号
   syscall.h 里 enum SYSCALL_NR 追加：

   ```
   SYS_OPEN, SYS_CLOSE, SYS_LSEEK, SYS_MKDIR ... SYS_STAT, SYS_PS
   ```

2. 用户桩函数（syscall.c）
   每个函数一句封装，按参数个数选 `_syscall0~3` 宏：

   ```
   int32_t open(const char *path, uint8_t flags) { return _syscall2(SYS_OPEN, path, flags); }
   int32_t mkdir(const char *path)             { return _syscall1(SYS_MKDIR, path); }
   ...
   ```

3. 内核向量表绑定（syscall-init.c）

   ```
   syscall_table[SYS_OPEN]     = sys_open;
   syscall_table[SYS_MKDIR]    = sys_mkdir;
   ...
   syscall_table[SYS_PS]       = sys_ps;   // 简易进程列表
   ```

三、新增小亮点

- `sys_ps` 已在内核实现（thread.c）：遍历全部队列，对齐打印 pid/ppid/status/ticks/name，供 shell 的 `ps` 命令直接调用。
- 所有路径相关调用统一先 `make_clear_abs_path()` 把用户给的相对/含“..”路径洗成绝对路径，再传内核，保持“用户态做转换、内核只认绝对路径”原则。

### 15.4.5 路径解析转换

一、需求
shell 键入的可能是

- 相对路径：`ls dir1/../file`
- 带 `.` / `..` 的绝对路径：`/a/b/../c/./d`
  内核只认**干净绝对路径**，所以用户态必须提前“洗干净”再传系统调用。

二、核心算法
`make_clear_abs_path(path, final_path)` 两步走：

1. 拼绝对：若 `path` 不以 `/` 开头 → `getcwd()` + 拼接
2. 洗“..”/“.”：
   - 顺序扫描每层目录名
   - 遇到 `".."` → 把 `final_path` 最右一层弹掉（回父目录）
   - 遇到 `"."` → 跳过
   - 其它 → 正常追加
     全程原地修改，不申请额外内存。

三、关键代码剪影

```c
static void wash_path(char *old_abs, char *new_abs) {
    char name[MAX_FILE_NAME_LEN] = {0};
    char *sub = old_abs;
    new_abs[0] = '/'; new_abs[1] = 0;
    while ((sub = path_parse(sub, name))) {
        if (!strcmp("..", name)) {
            char *slash = strrchr(new_abs, '/');
            if (slash != new_abs) *slash = 0;   // 退一级
            else *(slash+1) = 0;               // 已到根
        } else if (strcmp(".", name)) {        // 非“.”
            if (strcmp(new_abs, "/")) strcat(new_abs, "/");
            strcat(new_abs, name);
        }                                     // 是“.”直接忽略
    }
}
```

四、使用场景
shell 里任意命令在真正调 `open/stat/mkdir...` 之前都先：

```c
make_clear_abs_path(argv[i], final_path);
```

保证传给内核的路径**无“.”“..”**、**绝对**、**以单'/'开头**。

### 15.4.6 实现 ls、cd、mkdir、ps、rm 等命令

一、设计共识  
- 全部做成 **内建命令**（buildin_*），不依赖外部可执行文件，快速让 shell “能动”。  
- 统一接口：`buildin_xxx(uint32_t argc, char **argv)`  
- 路径先 `make_clear_abs_path()` → 再系统调用 → 结果即时 `printf`。

二、各命令要点速览  

| 命令      | 关键动作                                                     | 备注                                  |
| --------- | ------------------------------------------------------------ | ------------------------------------- |
| **ls**    | `stat()` 判文件类型；目录则 `opendir/readdir` 循环打印；支持 `-l` 长格式 `-h` 帮助 | 默认“当前目录”；总显示 `.` / `..`     |
| **cd**    | 调 `chdir()` 成功 → 把新路径写回 `cwd_cache` → 下次提示符实时更新 | 仅支持单参数；无参回根                |
| **mkdir** | `make_clear_abs_path()` → `mkdir()`                          | 拒绝创建根目录                        |
| **rmdir** | 同上 → `rmdir()`                                             | 空目录才能删；非空报错                |
| **rm**    | 同上 → `unlink()`                                            | 只删普通文件                          |
| **pwd**   | `getcwd()` → 打印                                            | 无参数                                |
| **ps**    | 直接 `ps()` 系统调用                                         | 固定表头：PID PPID STAT TICKS COMMAND |
| **clear** | `clear()`                                                    | 清屏并重置光标                        |

三、shell 路由表（cmd_execute）

## 15.5 加载用户进程

### 15.5.1 实现 exec

一、目标  
提供 `execv(path, argv)` ——**当前进程壳不换，内容全替换**：  
- 代码段 / 数据段 / 堆 / 栈 重新加载指定 ELF  
- 保持原 PID、打开文件、工作目录等上下文  
- 成功永不返回；失败返 -1

二、关键数据结构  
```c
// ELF 头
struct Elf32_Ehdr { e_ident[16], e_type, e_machine, e_entry, e_phoff, e_phnum ... };
// 程序头（段描述符）
struct Elf32_Phdr { p_type, p_offset, p_vaddr, p_filesz, p_memsz, p_flags ... };
```
只关心 `p_type == PT_LOAD` 的段

三、实现流程（3 层函数）  
1. `load(path)` —— 从文件系统加载 ELF  
   - 校验 ELF 魔数 `0x7F "ELF"` & 32-bit 小端  
   - 遍历所有 `PT_LOAD` 段：  
     - `segment_load(fd, offset, filesz, vaddr)`  
       ├ 计算需占页数 → 逐页 `get_a_page()`  
       ├ `sys_lseek` + `sys_read` 把文件内容拷到虚拟 `vaddr`  
       └ 若页已存在（旧进程映射）直接覆盖  
   - 返回 `e_entry`（入口虚拟地址）

2. `sys_execv(path, argv)` —— 替换当前进程  
   - 统计 argc  
   - `entry = load(path)`；失败返 -1  
   - 改当前 PCB 名称为新路径  
   - 伪造 **intr_exit** 现场：  
     ```
     intr_stack->eip = entry
     intr_stack->esp = 0xc0000000   // 用户栈顶
     intr_stack->ebx = argv         // 参数地址
     intr_stack->ecx = argc         // 参数个数
     ```
   - 汇编切换：`movl intr_stack, %esp; jmp intr_exit`  
     → 调度器下次选本进程时，从 `entry` 开始跑，参数已就位

四、失败回滚  
`load` 里任何一步错 goto `done` → `sys_close(fd)` → 返 -1，原进程无损。

### 15.5.2 让 shell 支持外部命令

一、目标  
shell 遇到**非内建**命令时，不再打印 “external command”，而是  
`fork() → 子进程 execv() → 父进程 wait()`  
实现真正“从磁盘加载任意用户程序”。

二、实现策略（修改 shell.c）  
1. 命令解析后，若 `argv[0]` 与所有内建都不匹配 → 走外部分支  
2. 外部分支模板  
```c
int32_t pid = fork();
if (pid) {                      // 父进程
    int32_t status;
    int32_t child_pid = wait(&status);   // 阻塞直到子进程 exit
    printf("child_pid %d status %d\n", child_pid, status);
} else {                        // 子进程
    make_clear_abs_path(argv[0], final_path); // 转绝对路径
    argv[0] = final_path;
    if (stat(argv[0], &file_stat) == -1) {
        printf("my_shell: %s: No such file\n", argv[0]);
        exit(-1);               // 错误立即退出
    }
    execv(argv[0], argv);       // 加载外部程序
    // 执行不到这里
}
```
3. 参数传递  
   - `argv[]` 已由 `cmd_parse()` 拆分好，直接交给 `execv`  
   - `execv` 内部把 `argc/argv` 塞进 intr_stack 的 `ecx/ebx`，用户程序 `_start` 取到后 `call main`

三、细节保护  
- 父进程必须先 `wait()`，否则会比子进程早进入下一轮主循环，把全局 `final_path` 等清空，子进程 `execv` 时路径失效。  
- 子进程路径不存在 → 立即 `exit(-1)` 让父进程 `wait` 收到状态 -1，不僵死。

四、运行快照  
```
[rabbit@localhost /]$ ./prog_no_arg
prog_no_arg from disk
child_pid 4 status 0
```
磁盘任意 ELF 现可被 shell 透明加载并回收。

一、目标  
把真正的用户 ELF 文件**先写入镜像 → 再当外部命令执行**，验证“shell + exec + 文件系统”全链路。

二、三步流程  
1. **宿主机侧**：  
   - 在 `command/` 写源码 `prog_no_arg.c`（仅 `printf("prog_no_arg from disk\n"); while(1);`）

- 裸盘写入仅一次，后续通过文件系统 API 拷贝，不破坏 hd80M.img 上既有元数据。

### 15.5.4 使用户进程支持参数

一、目标  
让 `main(int argc, char **argv)` 真正收到 shell 传来的参数，解决：  
1. 外部程序启动**前无栈** → 参数无法按普通 C 约定压栈；  
2. 程序名、选项、文件名等需从父进程 `execv(path, argv)` 安全传递到子进程用户空间。

二、整体方案  
“父进程准备 → 内核 intr_stack 暂存 → 用户 _start 取参 → 调用 main”

三、步骤拆解  

1. **shell 侧**  
   ```c
   char *argv[] = {"prog_arg", "/dir/file", NULL};
   execv("/prog_arg", argv);          // 路径已洗成绝对
   ```

2. **sys_execv 内核层**  
   - 统计 `argc`  
   - 把 `argv[]` 各指针、字符串**深度拷贝**到**新用户栈顶**（0xC0000000 往下）  
   - 在 intr_stack 里伪造：  
     ```
     ebx = 用户栈地址(argv 指针数组)
     ecx = argc
     eip = 入口
     esp = 新用户栈顶
     ```

3. **用户启动文件 `start.S`**（简易 CRT）  
   ```asm
   global _start
   _start:
       push ebx        # argv
       push ecx        # argc
       call main       # 进入用户 main
       push eax        # 收集 main 返回值
       call exit       # 自动交回内核
   ```

4. **用户程序**  
   ```c
   int main(int argc, char **argv){
       for(int i = 0; i < argc; i++)
           printf("argv[%d]=%s\n", i, argv[i]);
       return 66;      # 会被 exit 交给父进程 wait
   }
   ```

四、运行快照  
```
[rabbit@localhost /]$ ./prog_arg /dir/file
argv[0]=prog_arg
argv[1]=/dir/file
child_pid 4 status 66
```

execv 把参数深拷到用户栈顶 → 启动文件按 C 约定压栈 → main 收到 argc/argv，返回值经 exit→wait 回到父进程，**参数+返回值全链路打通**。

## 15.6 实现系统调用 wait 和 exit

### 15.6.1 wait 和 exit 的作用

一、exit 的使命  
1. 主动结束当前进程，把返回值交给内核  
2. 释放除 PCB 外的所有资源（页表、内存、打开文件）  
3. 把“遗言”exit_status 填进 PCB，供父进程领取  
4. 唤醒正在 wait 的父进程；若父已死，过继给 init

二、wait 的使命  
1. 阻塞父进程直到**任意一个**子进程调用 exit  
2. 取得子进程的 exit_status，内核随后回收子 PCB  
3. 防止“僵尸”——只留 PCB 不占资源，却占 PID  
4. 天然实现父子**执行顺序同步**（父等子干完再跑）

三、成对使用场景  
```
父进程              子进程
fork() ───────────────→
wait(&status)         do_work()
 ←── 阻塞 ───          exit(66)
status == 66  ←─── 回收/唤醒
```
无需空转 delay，CPU 不浪费，输出也不会交错。

exit 是子进程“交遗言+自焚”，wait 是父进程“收尸+点火”，二者成对出现，**同步执行流+回收资源+消灭僵尸**。

### 15.6.2 孤儿进程和僵尸进程

一、孤儿进程（ORPHAN）  
- **定义**：父进程先 exit，子进程仍在运行。  
- **处置**：内核把孤儿过继给 init（pid=1），由 init 负责后续 wait() 收尸。  
- **危害**：无，系统会自动重新指定父进程，资源最终可被回收。

二、僵尸进程（ZOMBIE）  
- **定义**：子进程已 exit 并提交返回值，但父进程**未** wait()，内核仍保留其 PCB 与退出码。  
- **特征**：  
  – 不占内存/文件等资源，只消耗一个 PID 表项；  
  – ps 显示 STAT=Z，PID 被持续占用。  
- **危害**：大量僵尸会耗尽 PID，导致后续 fork() 失败。  
- **解决**：kill 父进程 → 内核自动把僵尸子过继给 init → init wait() 回收。

三、对比速览  
| 项目       | 孤儿      | 僵尸                 |
| ---------- | --------- | -------------------- |
| 父进程状态 | 已死      | 活着且未 wait        |
| 子进程体   | 仍在运行  | 已销毁，仅 PCB 残留  |
| 资源占用   | 正常      | 只耗 PID             |
| 自动处理   | 过继 init | 需父进程或 init wait |

孤儿是被“放养”的娃，init 会领养；僵尸是“死而不埋”的娃，必须父进程（或 init）调用 wait() 来埋，否则 PID 会被啃光。

### 15.6.3 一些基础代码

一、PCB 增补

    int8_t exit_status;   // 保存子进程返回值

- `exit()` 把返回值写进这里

二、PID 资源池（bitmap 版）

- `allocate_pid()` / `release_pid()` 带锁，支持 fork/exit 反复分配回收

### 15.6.4 实现 wait 和 exit

一、接口原型  
```c
pid_t wait(int *status);          // 父进程调用
void exit(int status);            // 子进程调用
```

二、核心数据结构  
- `task_struct.exit_status`   – 保存返回值  
- `TASK_HANGING`             – 子进程已退出但 PCB 未回收  
- `TASK_WAITING`             – 父进程阻塞等待子退出

三、exit 流程（sys_exit）  
1. 当前进程 `exit_status = status`  
2. 把所有子进程过继给 init（`parent_pid = 1`）  
3. `release_prog_resource()` – 释放页表、用户内存、关闭文件  
4. 若父进程正处于 `TASK_WAITING` → `thread_unblock(parent)`  
5. 自己设为 `TASK_HANGING` 并主动 `schedule()` 永不返回

四、wait 流程（sys_wait）  
```c
while(1){
    // ① 优先收割已死子进程
    child = list_traversal(..., find_hanging_child, ppid);
    if(child){
        *status = child->exit_status;
        pid = child->pid;
        thread_exit(child, false);   // 回收 PCB
        return pid;
    }
    // ② 若无子进程，返回 -1
    if(无子) return -1;
    // ③ 有子但全活着，阻塞自己
    thread_block(TASK_WAITING);      // 直到某子 exit 唤醒
}
```

五、成对效果  
- 子 `exit(66)` → 父 `wait(&st)` 得 `st==66` 且 PCB 被回收  
- 父若提前死 → 子变孤儿，由 init 负责后续 wait  
- 父永不 wait → 子成僵尸，PID 被占，需杀父才能释放  
  `exit` 负责“交遗言+自挂”；`wait` 负责“阻塞收尸+清 PCB”——二者配对，完成返回值传递、资源彻底回收、僵尸消灭。

## 15.7 管道

### 15.7.1 管道的原理

一、概念  
管道 = **内核里的一段环形缓冲区**，两端各给一个 fd：  
- fd[0] 只读  
- fd[1] 只写  
数据单向流动，先进先出，读后即删。

二、为什么要有管道  
进程相互独立，需要**低成本单向数据流**；磁盘文件太慢，共享内存复杂，管道折中：  
- 驻留内存，速度 ≈ 内存拷贝  
- 复用已有 read/write 接口，无需新 API  
- 天然同步：空则读阻塞，满则写阻塞

三、匿名管道生命周期  
1. 创建：  
   `int fd[2]; pipe(fd);`              // 内核分配环形缓冲区 + 两个 fd  
2. fork：  
   子进程**继承**两个 fd，父子同时拥有读写端  
3. 关闭无用端：  
   父关 fd[0]，子关 fd[1] → 形成 **父→写→管道→读→子** 单向通道  
4. 写完 → 父关 fd[1] → 子 read 返回 0（EOF）  
5. 全部关闭 → 内核回收缓冲区

四、环形缓冲区机制（ioqueue 复用）  
- 头指针：写位置  
- 尾指针：读位置  
- 空：头 == 尾 → 读者睡眠  
- 满：头 == 尾-1 → 写者睡眠  
- 每次 read/write 更新指针并唤醒对方

五、容量与限制  
本系统：单页 4 KB（减 1 字节做满/空判别）  
→ 最大无损传输 4095 字节；更大数据需分批读写。

管道就是**内核环形缓冲区 + 两个文件描述符**，利用 fork 继承后“各关一端”形成单向数据流，既复用 read/write 又自带阻塞同步。

### 15.7.2 管道的设计

一、总体思路  
“复用现有文件系统框架，零新增结构”——把管道伪装成一个**特殊文件**：

二、文件描述符三级链  
```
PCB.fd_table[]
   ↓ 下标
file_table[]
   ↓ 指针
struct file {
    uint32_t fd_pos;   → 复用为“管道引用计数”
    uint32_t fd_flag;  → 复用为 PIPE_FLAG（0xFFFF）
    struct inode *fd_inode; → 指向**内核环形缓冲区页**
}
```
父子进程只要共享同一个 `file_table[]` 项，就共享同一管道缓冲。

三、环形缓冲区管理  
- 一页内核空间（4 KB）做 ioqueue  
- `pipe_read()` / `pipe_write()` 内部：  
  – 读空→读者阻塞（TASK_BLOCKED）并唤醒写者  
  – 写满→写者阻塞并唤醒读者  
- 每次只读写“当前可处理量”，避免 shell 多级管道时永久睡眠（见下限制）。

四、引用计数与生命周期  
- `fd_pos` 初始 = 2（pipe 返回两个 fd）  
- fork 后 `fd_pos++`（子继承）  
- 每次 close 对应 `fd_pos--`；减到 0 释放缓冲页 → 自动销毁。

五、容量与折中限制  
单页 4095 有效字节；shell 多级管道（a|b|c）**串行执行**，未并行：  
→ 写满或读空会导致**永久阻塞**，故读写均取“剩余空间/当前数据量”最小值，**分批传输**，牺牲带宽换简单实现。


管道设计 = “把环形缓冲区页塞进 file 结构，用 fd_flag 打标记、fd_pos 当引用计”，父子共享同一 file_table 项即可单向通信，生命周期靠引用计数归零自动释放。

### 15.7.3 管道的实现

一、系统调用原型  
```c
int32_t pipe(int32_t pipefd[2]);   // 成功返回 0，失败 -1
```
pipefd[0] 只读，pipefd[1] 只写，父子关闭无用端后即形成单向数据流。

二、关键函数与实现步骤  

| 函数                       | 作用     | 实现要点                                                     |
| -------------------------- | -------- | ------------------------------------------------------------ |
| `sys_pipe(pipefd)`         | 创建管道 | 1. 取空闲 `file_table[]` 项 `gfd`<br>2. 申请 1 页内核空间 → 初始化为 `ioqueue`<br>3. `fd_flag = PIPE_FLAG`（0xFFFF），`fd_pos = 2`（引用计数）<br>4. 两次 `pcb_fd_install(gfd)` 得到 r/w fd，填回 `pipefd[0/1]` |
| `pipe_read(fd, buf, cnt)`  | 读管道   | 1. 取对应 `ioqueue` 指针<br>2. 可读长度 = `min(cnt, ioq_length())`<br>3. 循环 `ioq_getchar()` 拷贝到用户 buf<br>4. 若空则阻塞自身并唤醒写者 |
| `pipe_write(fd, buf, cnt)` | 写管道   | 1. 可写长度 = `min(cnt, bufsize - ioq_length())`<br>2. 循环 `ioq_putchar()` 拷贝自用户 buf<br>3. 若满则阻塞自身并唤醒读者 |

三、引用计数与生命周期  
- `fork` 时 `update_inode_open_cnts()` 发现 `is_pipe(fd)` → 仅 `file_table[gfd].fd_pos++`（不碰 inode 引用）  
- `close` 时 `--fd_pos`；减至 0 → `mfree_page` 释放环形缓冲页，文件结构清零。

四、阻塞/唤醒机制  
使用 `ioqueue` 自带  
- `ioq_getchar()` 空 → `thread_block(TASK_BLOCKED)` 并 `thread_unblock(ioq->consumer)`  
- `ioq_putchar()` 满 → `thread_block(TASK_BLOCKED)` 并 `thread_unblock(ioq->producer)`  
保证数据不丢、CPU 不空转。


管道实现 = “申请一页 ioqueue → 塞进 file_table → 用 fd_flag/fd_pos 做标记+引用计数 → read/write 只操作环形缓冲区并配合阻塞唤醒”，父子共享同一 file 项即完成单向无损数据流。

### 15.7.4 利用管道实现进程间通信

一、实验目标  
用新建的系统调用 `pipe + fork + read/write` 写个**最小 IPC 演示**：  
父进程写一句 → 子进程读一句 → 各自 exit 并带不同返回值，父 wait 回收。

二、用户程序骨架（prog_pipe.c）  
```c
int main(int argc, char **argv) {
    int32_t fd[2];
    pipe(fd);                       // fd[0]读 fd[1]写

    int32_t pid = fork();
    if (pid) {                      // 父：只写
        close(fd[0]);               // 关闭读端
        write(fd[1], "Hi, my son, I love you!", 24);
        close(fd[1]);
        return 8;                   // 父退出码
    } else {                        // 子：只读
        close(fd[1]);               // 关闭写端
        char buf[32] = {0};
        read(fd[0], buf, 24);       // 阻塞直到有数据
        printf("I`m child, father said: \"%s\"\n", buf);
        close(fd[0]);
        return 9;                   // 子退出码
    }
}
```

三、运行流程  
1. `pipe` 创建环形缓冲 → 返回 r/w fd  
2. `fork` 后父子**共享同一 file_table 项**（引用计数=2）  
3. 各自关闭无用端 → 引用计数=1，形成**单向通道**  
4. 父 write 唤醒子，子 read 完成后两者分别 exit  
5. 父进程 `wait(&status)` 先后收集 8、9 两个返回值并打印

四、快照  
```
$ ./prog_pipe
I`m child, father said: "Hi, my son, I love you!"
[父shell] child_pid 2 status 8
[init]    child_pid 5 status 9
```
status 与 return 值一致，父子通信+回收双验证通过。

“pipe → fork → 各关一端 → write/read → exit → wait”六连击，首次验证**内核环形缓冲区 + 文件描述符共享**的父子进程单向通信完整闭环。

### 15.7.5 在 shell 中支持管道

一、目标  
让 shell 解析并执行 **多级管道命令**：  
`cmd1 | cmd2 | ... | cmdn`  
前一条命令的 stdout 直接作为后一条命令的 stdin，实现“命令流水线”。

二、核心机制  
1. **输入/输出重定向**  
   `fd_redirect(old_fd, new_fd)` 把 PCB.fd_table[old_fd] 换成 new_fd 对应的全局 file 下标，即可把 stdout(1) 或 stdin(0) 指向任意打开文件（含管道）。  
2. **每级命令独享一对管道**  
   循环：  
   - 创建 pipe fd[2]  
   - 父（writer）重定向 stdout → fd[1]  
   - 子（reader）重定向 stdin  → fd[0]  
   - 执行完关闭当前 pipe，进入下一轮

三、执行六步走（my_shell 伪码）  
```
① strchr(cmd_line,'|') 找到第一个 '|'
② pipe(fd)                    // 新建环形缓冲
③ fd_redirect(1, fd[1])       // 本级 stdout → 管道写
④ 解析并执行 cmd1             // 输出落入缓冲区
⑤ fd_redirect(0, fd[0])       // 下级 stdin ← 管道读
⑥ 循环 ③-⑤ 直到无 '|'；最后一级 stdout 重定向回屏幕(1)
⑦ fd_redirect(0,0); fd_redirect(1,1); close(fd[0]); close(fd[1])
```

四、简化实现细节  
- 命令串行执行：前一命令 exit 后才启动下一命令（未并行，避免永久阻塞）  
- 每次读写取 **min(请求字节, 缓冲剩余/已用)**，保证不堵死  
- 全局数组 `final_path` 存放各级清洗后的绝对路径，单控制台无并发覆盖问题

五、运行示例  
```
[rabbit@localhost /]$ cat file1 | cat
hello,world          ← file1 内容经两次 cat 仍正常输出
child_pid 2 status 0
```
多级管道验证通过。

shell 通过“**每级 pipe + 重定向 stdin/stdout + 串行执行**”实现命令流水线，虽简化为单线程顺序，但已完整展示“|”符号背后的输入输出重定向与内核环形缓冲区复用机制。
