# 第 9 章 线程

## 9.1 实现内核线程

### 9.1.1 执行流

​	早期计算机是单任务系统，任务必须串行执行。若一个任务（如任务A）执行时间很长（如一天），其他短任务（如任务B）必须等待，导致资源利用率低和用户体验差。

​	通过“多道程序设计”实现伪并行：即使只有一个处理器，也可以通过快速切换任务，让用户感觉多个任务在同时运行。任务切换由**任务调度器**完成，它是操作系统中的一个软件模块，负责管理任务表（进程表/线程表），并按算法（如时间片轮转）选择任务上处理器执行。**伪并行：**任务切换本身消耗处理器时间，因此所有任务的总执行时间比串行执行更长。但优点在于提高了响应速度，让短任务不必长时间等待。

**执行流**是一段逻辑上独立的指令序列，是处理器执行的基本单元。每个执行流都有独立的上下文环境：包括自己的栈、寄存器映像和内存资源。执行流可以是大到整个进程，小到一个函数（线程本质上就是函数）。任何代码块都可以成为独立的执行流，只要为其准备好所需的上下文环境（寄存器、栈等）。

**任务与调度：**任务是人为主观划分的执行单元，调度器只认执行流作为调度单位。任务切换的本质是改变处理器中程序计数器（PC）的指向，从而改变执行流。

### 9.1.2 线程到底是什么

从代码看线程是“运行函数的另一种方式”a

![image-20250919151807521](F:\OS\xuexi\truth\图片\image-20250919151807521.png)



线程就是**让函数单独上 CPU 运行的机制**。普通函数是“顺带执行”，线程是“专门执行”。

**线程的本质：独立执行流（Execution Flow）**线程是一套机制，此机制可以为一般的代码块创造它所依赖的上下文环境，从而让代码块具有独立性，因此在原理上线程能使一段函数成为调度单元（或称为执行流），使函数能被调度器“认可”，从而能够被专门调度到处理器上执行。这样，函数就可以被加入到线程表中作为调度器的调度单元，从而有机会单独获得处理器资源，也就是说，处理器不是把线程中调用的函数和其他指令混在一块执行的，或者说不是在执行整个进程时顺便执行了该函数，而是单独、专门执行了此函数。

**普通函数调用**：函数依附于所在线程（或进程）的执行流。被“顺带”执行，调度器看不到它。可能时间片用完还没执行到。

**线程中调用函数**：函数成为独立执行流，被调度器“看见”。单独获得 CPU 时间片，**专门执行**。

**线程与执行流的关系：线程就是执行流。**执行流 = 独立上下文 + 可调度**。**线程机制（如 pthread）就是为函数创建独立上下文的工具。

### 9.1.3 进程与线程的关系、区别简述

| 概念 | 定义                       | 本质                                             |
| ---- | -------------------------- | ------------------------------------------------ |
| 程序 | 静态的指令集合，躺在磁盘上 | 被动数据                                         |
| 进程 | 程序运行时的一个“实例”     | 运行中的程序，拥有资源（地址空间、栈、寄存器等） |
| 线程 | 进程内部的“执行流”         | 真正上 CPU 跑代码的单元                          |

进程 = 资源 + 线程  进程是“容器”，提供地址空间、页表、打开文件、全局变量等资源。线程是“执行实体”，使用进程的资源，真正完成计算工作。

**其区别是**：

| 项目     | 进程                       | 线程                         |
| -------- | -------------------------- | ---------------------------- |
| 地址空间 | 独立页表，完整虚拟地址空间 | 共享进程地址空间，无独立页表 |
| 创建开销 | 需分配页表、初始化资源     | 只需创建 PCB、栈、寄存器映像 |
| 切换开销 | 需切换页表、刷新 TLB       | 共享地址空间，无需刷新 TLB   |
| 通信方式 | 需 IPC（管道、共享内存等） | 直接读写共享变量，通信零成本 |

线程与进程**调度单元**：都是“执行流”，由调度器按相同规则调度。**上下文**都需保存/恢复寄存器、栈指针等。**区别**在于进程切换伴随**地址空间切换**（页表切换）。线程切换**地址空间不变**，仅切换执行上下文。

**线程不能独立存在**，必须依附于进程。**线程无资源**，所有资源（代码、数据、堆、打开文件）均来自进程。**线程间共享一切**，因此：通信高效，但需同步机制防止数据竞争。

**进程**由操作系统创建，分配资源，至少有一个线程（主线程），所有线程结束后，资源回收。

**线程**由进程（或内核）创建，复用进程资源，可创建多个线程并发执行，单独线程结束不影响其他线程（内核线程）。 

**单线程进程**是传统意义上的“进程”。只有一个执行流，即主线程。

**多线程进程**：同一地址空间内有多个执行流。例如：浏览器（UI 线程、网络线程、渲染线程等）。

### 9.1.4 进程与线程的状态

真正被调度器拎来拎去的是线程——也就是那一条条“执行流”。只要 CPU 不罢工，任何时刻它只能跑一条执行流，其它流要想继续就得排队，于是操作系统给每条流分了三种状态：

1. 运行态——正在 CPU 上运行；  
2. 就绪态——所有资源都 OK，只等调度器调度；  
3. 阻塞态——等 I/O、等信号、等用户输入，总之在等外部资源，暂时没资格上 CPU。

这三种状态不是写死就完事，而是会随着“中断到来”“数据读完”“时间片用光”这些事件来回切换。调度器当CPU来，先看谁就绪，再让运行的程序下cpu，最后把就绪的塞进去。整个过程对线程来说是被动的，它只负责把状态一改，切换就完成。于是“状态”成了调度器手里最顺手的调度依据，而 PCB 里必须留一个字节给这个状态位，否则它就不知道下一步该谁。

### 9.1.5 进程的身份证——PCB

线程也是轻量级进程：内核眼里没有进程 线程区分，只有“任务”——一个带着上下文的包裹。包裹里装啥？就是 PCB（task_struct）。

寄存器映像：相当于“现场照片”。时钟中断一来，CPU 把当前所有寄存器压进内核栈，栈指针再被塞进 PCB->self_kstack，回头调度回来时能原样恢复。  

内核栈指针：告诉内核该进程的 0 级栈顶在哪，中断嵌套或系统调用时别跑别的栈里去。  

状态位：就是上一节的三张标签，调度器每次调度都要先看状态。  

时间片 ticks & 优先级 priority：ticks 每次时钟中断减 1，减到 0 就下cpu，priority 给 ticks 重新加时，优先级越高时间越长。  

页表指针 pgdir：线程共享进程地址空间，所以线程这一项填 NULL；真正的进程才有自己独立的页目录地址。  

名字/魔数：调试时一眼认出进程是谁，魔数（如 0x19870916）用来检测栈溢出——如果被踩成 0x19870915，内核立刻报警越界。

PCB 就是操作系统给每个任务的识别表，只要调度器想换进程，先把旧进程的现场保存到 CPU 上，再把新的进程换上，任务就无缝交接。没有这pcb，线程连“我是谁、我在哪”都回答不了，更谈不上被调度。

### 9.1.6 实现线程的两种方式——内核 vs 用户

**在用户空间中实现线程**  

 特权级，进程特权级。 进程自己对其进行调度，内核完全不知。  

优点：线程的调度算法是由用户程序自己实现的，可以根据实现应用情况为某些线程加权调度；切换不用进内核，速度飞快；移植性棒，连 DOS 都能跑。  

缺点：  
1. 内核只看到“进程”整体，一旦某个用户线程执行 堵了，内核直接把整个进程挂起，其它用户线程再就绪也不能运行。  
2. 内核给进程的时间片被用户调度器再细分，但内核不会主动打断“正在跑”的用户线程，只要它不让出时间片，兄弟线程也无法运行。  
3. 内核调度单位是进程，所以哪怕 16 核，你的进程也只被丢到一个核上，用户线程也能利用一部分资源，实际上这反而抵销了内部调度带来的提速。

**在内核空间中实现线程**  

0 特权级，内核空间。 操作系统统一调度，所有线程无论属于谁，都在同一张“就绪队列”里。  

缺点：每次切换要陷入内核，保存/恢复现场比用户级多几条指令，开销略大。  

优点：  
1. 真并行——16 核就能同时跑 16 个线程大大提速。  
2. 阻塞隔离——一个线程读磁盘堵了，内核只把它自己单独提出 CPU，其它线程继续运行。  
3. 公平抢占——时钟中断一视同仁，时间片到了就换人，没人能一直占用。

我们要的是“提速 + 隔离 + 多核”，那点内核进出开销算不了什么，故使用“内核级线程”路线。

1. PCB 最小原型 `task_struct`  

   `self_kstack`：指向本线程 0 级栈顶，固定放在结构体偏移 0 处，方便汇编直接存取。  

   `status / priority / name / stack_magic`：状态、优先级、调试名、栈溢出魔数（0x19870916）。  

   整个 PCB 占一页（4 KiB），页低地址开始是结构体，高地址向下生长为内核栈。

2. 双栈结构  

   `intr_stack`：中断进入时硬件/统一入口自动压入的完整上下文，共 36 字节，位于栈页最顶端。  

   `thread_stack`：线程首次被调度或 `switch_to` 切换时使用的软件上下文，仅保存 ABI 规定需 callee 保护的 4 个寄存器（ebp, ebx, edi, esi）以及返回地址、线程入口参数。

3. 入口约定  

   `kernel_thread` 不是被 call，而是被 ret“弹”入，因此 `thread_stack` 里先压占位返回地址，再压 `function/func_arg`，形成符合 C 调用规范的栈帧，保证 `kernel_thread(function, func_arg)` 能正确取参并继续调用用户函数。

4. 初始化顺序  

   `init_thread` 清零 PCB → 填写属性 → 把 `self_kstack` 指向页顶；  
   `thread_create` 先后预留 `intr_stack` 与 `thread_stack` 空间 → 按布局填入 `kernel_thread` 入口及参数 → 初始化 4 个寄存器为 0。

    self_kstack 是各线程的内核栈顶指针，当线程被创建时，self_kstack 被初始化为自己 PCB 所在页的顶端。之后在运行时，在被换下处理器前，我们会把线程的上下文信息（也就是寄存器映像）保存在 0 特权级栈中，self_kstack 便用来记录 0 特权级栈在保存线程上下文后的新的栈顶，在下一次此线程又被调度到处理器上时，可以把 self_kstack 的值加载到 esp 寄存器，这样便从 0 特权级栈中获取了线程上下文，从而可以加载到处理器中运行。



### 9.2.2 线程的实现

总入口 `thread_start(name, prio, function, func_arg)`  
1. 申请物理页：`get_kernel_pages(1)` 返回页基址，直接作为 `struct task_struct *thread`。  
2. 初始化 PCB：`init_thread(thread, name, prio)`  
   
   整页清零 → 拷名字 → 状态 `TASK_READY` → 优先级 `prio` → `ticks=prio` → `elapsed_ticks=0` → `pgdir=NULL`（线程无独立页表）→ `stack_magic=0x19870916`。  
   
   `self_kstack` 初始指向页高地址 `(uint32_t)thread + 0x1000`，后续向下生长。  
3. 构造栈：`thread_create(thread, function, func_arg)`  
   a. 先向下预留 `sizeof(intr_stack)`——中断完整上下文空间。  
   b. 再向下预留 `sizeof(thread_stack)`——软件切换上下文空间。  
   c. 填充 `thread_stack` 字段：  
   
   `eip = kernel_thread`（首次执行地址）  
   
   `function / func_arg` 供 `kernel_thread` 调用  
   
   4 个被保存寄存器 `ebp/ebx/edi/esi = 0`  
   
   `unused_retaddr` 占位，保持 C 调用栈帧对齐  
   d. 最终 `self_kstack` 指向 `thread_stack` 底部，即下一次 `switch_to` 时的 esp 值。  

首次投入运行（临时跳板）  
由于尚未开启正式调度，直接在 `thread_start` 尾段嵌入汇编：  

```asm
movl %0, %%esp      // esp ← thread->self_kstack
pop %%ebp; pop %%ebx; pop %%edi; pop %%esi
ret                 // 弹出 eip = kernel_thread
```
4 个 pop 把 0 值装回寄存器，满足 ABI 要求的 callee-saved 初始状态。  ret 进入 `kernel_thread`，栈顶已布置好“返回地址+参数”，形成标准 C 调用帧。  

`kernel_thread(function, func_arg)` 内部  先 `intr_enable()` 打开中断，确保时钟中断可以到达，防止新线程独占 CPU。  执行 `function(func_arg)`，用户代码开始运行；函数返回后线程生命周期由后续调度器管理（当前实现无限循环，永不返回）。  

实验验证  
`main()` 中连续启动两个线程，分别打印不同字符串；主线程也循环打印。  
由于时钟中断已注册但尚未启用正式多线程调度，此时靠临时汇编跳板直接启动新线程，肉眼可见三行字符交替输出，验证：  

栈布局正确，线程能跑到用户函数；  中断开启后时钟滴答正常，后续可无缝接入正式调度器。

## 9.3 核心数据结构，双向链表

程序=算法+数据结构。

内核中也要用到队列，比如进程的就绪队列、锁的等待队列等，为了维护内核中的各种队列，本节要实现自己的链表—双向链表。

**设计目标**：

给线程就绪队列、锁等待队列等**公共基础设施**一个可重入、关中断保护的通用链表  

**不存储数据**——节点只保留前后指针，业务结构通过 `offset/elem2entry` 反算地址，实现“零拷贝”  

**固定头尾**——`head` 与 `tail` 永驻内存，插入/删除全部发生在二者之间，遍历方向可前可后  

**关键实现细节：**

1. **节点结构**  
  ```c
  struct list_elem { struct list_elem *prev, *next; };
  struct list { struct list_elem head, tail; };
  ```
  初始化后 `head.next = &tail; tail.prev = &head;`，空表判定仅需一句 `head.next == &tail`

2. **关中断保护**  
    所有修改型操作（insert/remove）全程 `intr_disable()` → 业务代码 → `intr_set_status(old)`，保证临界区原子性；后续可无缝替换为自旋锁

3. **offset/elem2entry 宏**  
  ```
  offset(type, member)  // 成员在结构体内的字节偏移
  elem2entry(type, member, ptr)  // 由成员地址反推外结构地址
  ```
  典型用法：  
  ```
  task = elem2entry(struct task_struct, general_tag, node);
  ```

4. **接口**  

`list_push` / `list_append` —— 头/尾插入，实现栈或 FIFO  

`list_remove` —— 任意节点 O(1) 摘除  

`list_pop` —— 弹出首节点，返回节点指针  

`list_traversal(func, arg)` —— 回调式遍历，命中即停，返回命中节点  

`list_empty` / `list_len` —— 空表判定与长度统计

**与线程调度的衔接：**

PCB 内嵌两个 `list_elem`：  
`general_tag` —— 就绪/阻塞等**功能性队列**  
`all_list_tag` —— 全局** bookkeeping 队列**，遍历全部任务  调度器只保存 `struct list` 头，通过 `elem2entry` 即可还原 PCB，无需额外内存管理

#### 3．实现任务切换函数 switch_to

任务调度是由时钟中断发起，由中断处理程序调用 switch_to 函数实现的，第一次被时钟中断打断时要保护一次上下文，之后时间中断跳转到switch所调换的任务时又要再次保护中断处理程序的上下文，共二次保护。

第一部分是进入中断时的保护，这保存的是任务的全部寄存器映像，也就是进入中断前任务所属第一层的状态，这些寄存器映像相当于任务中用户代码的上下文。

第二部分是保护内核环境上下文，根据 ABI，除 esp 外，只保护 esi、edi、ebx 和 ebp 这 4 个寄存器就够了。

## 总结：

第九章主要从零开始实现了一个**内核级线程机制**：先明白线程本质就是“能让函数单独上 CPU 的执行流”，再了解 PCB、双栈、就绪队列等基础数据结构，用汇编+C 写好线程创建、切换和调度器，最后把主线程和普通线程都塞进一条时间片轮转的调度链里，让多个线程真正交替跑起来。
