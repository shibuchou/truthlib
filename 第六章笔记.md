## 6.1 函数调用约定简介

### 什么是“函数调用约定”？
> **Calling Convention**，即**调用约定**，是**调用函数时的一套规则**，高级语言中不涉及，用于规范：
- **参数如何传递**（寄存器？栈？）

  在使用函数的过程中需要进行参数的传递，一般将参数存储在内存中，而在内存中栈是最适合用来存储参数的，优点是：

  （1）首先，每个进程都有自己的栈，这就是每个内存自己的专用内存空间。

  （2）其次，保存参数的内存地址不用再花精力维护，已经有栈机制来维护地址变化了，参数在栈中的位置可以通过栈顶的偏移量来得到。

- **参数传递顺序**（从左到右？从右到左？）

  c语言默认从右到左

- **谁来清理栈空间**（调用者？被调用者？）

  c语言默认调用者清理，stdcall约定为被调用者清理

- **哪些寄存器需要保存**（调用者保存？被调用者保存？）

### 为什么需要调用约定？
在**C语言**中，我们写函数调用时非常简单：
```c
int sub = subtract(3, 2);
```
但在**汇编层面**，必须明确：
- 参数3和2**放在哪里**？

- 函数返回后，**栈空间谁来清理**？

- 如果**多个程序员协作**，必须统一规则，否则无法正确传参和清理栈。

### 参数传递的两种方式
####  1. 寄存器传参
- **优点**：速度快（寄存器比内存快）
- **缺点**：
  - 寄存器数量有限（x86只有8个通用寄存器）
  - 函数调用前后需要**保存/恢复寄存器现场**
  - 如果函数嵌套调用，容易**覆盖寄存器值**

####  2. 栈传参（主流方式）
- **优点**：
  - 每个进程有**独立的栈空间**，不会互相干扰
  - 栈空间足够大，适合传递**任意数量的参数**
  - 栈指针（esp）自动维护，**无需手动管理地址**
- **缺点**：比寄存器稍慢（内存访问）

### 栈传参的两个关键问题

cdecl 调用约定由于起源于 C 语言，所以又称为 C 调用约定，是 C 语言默认的调用约定。

####  问题1：参数压栈顺序？
- **从右到左压栈**（C语言默认）cdecl约定
  
  例如：`subtract(3, 2)`，先压2，再压3
  
- **为什么？**

   方便被调用函数**从栈顶依次取参**（第一个参数在栈顶）

####  问题2：谁来清理栈空间？
- **调用者清理**（cdecl约定）c语言默认
- **被调用者清理**（stdcall约定）
- **区别**：
  - 如果函数参数数量**不固定**（如printf），必须由**调用者清理**（因为被调用者不知道参数数量）
  - 如果参数数量**固定**，可以由**被调用者清理**（更简洁）

### 调用约定的分类对比
| 约定名       | 参数传递顺序       | 栈清理方         | 适用场景                        |
| ------------ | ------------------ | ---------------- | ------------------------------- |
| **cdecl**    | 从右到左           | **调用者清理**   | C语言默认（支持变参函数）       |
| **stdcall**  | 从右到左           | **被调用者清理** | Win32 API（参数固定）           |
| **fastcall** | 前几个参数用寄存器 | 不定             | 优化速度（寄存器传参）          |
| **pascal**   | 从左到右           | 被调用者清理     | 早期Windows API（已淘汰）       |
| **thiscall** | 从右到左           | 不定             | C++成员函数（this指针特殊处理） |

### cdecl vs stdcall 的汇编对比
####  cdecl示例（调用者清理栈）
```asm
; 调用者
push 2          ; 压入参数b
push 3          ; 压入参数a
call subtract   ; 调用函数
add esp, 8      ; 调用者清理栈（2个参数×4字节=8字节）

; 被调用者
push ebp           ; 备份地址
mov ebp, esp
mov eax, [ebp+8]   ; 取参数a
sub eax, [ebp+12]  ; 减参数b
pop ebp
ret                ; 无参数，直接返回
```

####  stdcall示例（被调用者清理栈）
```asm
; 调用者
push 2
push 3
call subtract
; 无需清理栈！

; 被调用者
push ebp
mov ebp, esp
mov eax, [ebp+8]
sub eax, [ebp+12]
pop ebp
ret 8              ; 被调用者清理8字节栈空间
```

## 6.2汇编语言和c语言混合编程

汇编语言和 C 语言混合编程可分为两大类。

（1）单独的汇编代码文件与单独的 C 语言文件分别编译成目标文件后，一起链接成可执行程序。

（2）在 C 语言中嵌入汇编代码，直接编译生成可执行程序。

### 6.2.1 浅析 C 库函数与系统调用  

作用：用最小的例子说明“汇编代码如何像C那样调用内核服务”，为后面内核打印函数打基础。

#### 1. 系统调用概念回顾  
- 系统调用 = 用户态→内核态的“功能号+参数”请求。  用户程序没有权限，只能向操作系统寻求帮助，故系统调用是供用户程序来使用的，操作系统权利至高无上，不需要使用自己对外发布的功能接口，即系统调用。

  定义在/usr/include/asm/unistd.h 文件。

  如果不知道某个系统调用的用法，可以用 man 命令来查看，方法是 man 2 系统调用名。

- Linux 所有系统调用共用**0x80号中断**，子功能号放 **eax**。  

- 参数传递规则（x86 32位）：  
  - 参数 ≤5 个：依次放 ebx、ecx、edx、esi、edi；  
  - 参数 >5 个：把参数块首地址放 ebx。  

#### 2. 两种调用方式  
| 方式            | 特点                   | 实例                              |
| --------------- | ---------------------- | --------------------------------- |
| ① 走 libc       | 简单、可移植，但依赖库 | `write(1,"hi",3);`                |
| ② 直接 int 0x80 | 不依赖任何库，效率高   | 自己把参数塞进寄存器后 `int 0x80` |

（1）将系统调用指令封装为 c 库函数，通过库函数进行系统调用，操作简单。

（2）不依赖任何库函数，直接通过汇编指令 int 与操作系统通信。

#### 3. 示例代码 syscall_write.S  

功能：先“模拟”libc的write，再由汇编自己发系统调用。

![image-20250809204258510](F:\OS\xuexi\truth\图片\image-20250809204258510.png)

- **方式1（模拟libc）**  
  ```asm
  push len
  push msg
  push fd
  call simu_write     ; 自己写的仿 write 函数
  add esp,12
  ```
  `simu_write` 内部其实就是把栈中参数搬到寄存器后再 `int 0x80`。

- **方式2（直接系统调用）**  
  ```asm
  mov eax,4      ; 功能号 __NR_write
  mov ebx,1      ; fd=1 (stdout)
  mov ecx,msg
  mov edx,len
  int 0x80
  ```

- **退出**  
  ```asm
  mov eax,1      ; 功能号 __NR_exit
  int 0x80
  ```

#### 4. 编译链接流程  
```bash
nasm -f elf -o syscall_write.o syscall_write.S
ld   -o syscall_write.bin syscall_write.o
chmod u+x syscall_write.bin
```
运行效果：终端先打印 `c library says: hello world!`，再打印 `syscall says: hello world!`。

### 6.2.2 汇编语言与C语言共同协作  
作用：演示 **“汇编 ↔ C”互调** 的最小闭环，为后面把打印函数拆成汇编实现做准备。

#### 1. 协作模型  
- 汇编文件与C文件各自独立编译成 `.o`，再链接。  
- 汇编器(nasm)与C编译器(gcc)通过 **符号表 + 调用约定** 沟通。  
  - 导出符号：`global 符号名`（nasm）/ `extern 原型`（C）。  
  - 符号解析：链接器负责把“声明”与“定义”绑在一起。

#### 2. 示例文件  
| 文件         | 角色   | 关键代码                                                     | 说明                                                        |
| ------------ | ------ | ------------------------------------------------------------ | ----------------------------------------------------------- |
| C_with_S_c.c | C端    | `extern void asm_print(char*,int);` <br> `c_print→asm_print` | C 代码被汇编调用，又去调用汇编                              |
| C_with_S_S.S | 汇编端 | `extern c_print` <br> `_start→c_print` <br> `asm_print→int 0x80` | 汇编先执行，调用 C 的 `c_print`，C 再调用汇编的 `asm_print` |

#### 3. 编译链接  
```bash
nasm -f elf C_with_S_S.S -o C_with_S_S.o
gcc  -m32 -c C_with_S_c.c -o C_with_S_c.o
ld   -m elf_i386 -o c_asm.bin C_with_S_S.o C_with_S_c.o
```
运行结果：屏幕打印  
```
asm_print says hello world!
```
（由 `asm_print` 通过系统调用 write 输出）。

在汇编代码中导出符号供外部引用是用的关键字 global，引用外部文件的符号是用的关键字extern。

y在 C 代码中只要将符号定义为全局便可以被外部引用（一般情况下无需用额外关键字修饰，具体请参考 C 语言手册），引用外部符号时用 extern 声明即可。

## 6.3 实现自己的打印函数

### 6.3.1 显卡的端口控制  

#### 目的  
彻底摆脱 BIOS 中断或 Linux 系统调用，**直接操纵显卡**在 80×25 文本模式下输出。  之前对显卡的操作不过是简单的mov，现在将和操作硬盘一样，使用端口进行操作。

#### 核心知识点  
1. **显存布局**  
   
   - 起始物理地址 `0xB8000`，每个字符占 2 B：低字节 ASCII，高字节属性。  
   - 显卡的寄存器很多，寄存器的地址范围是 0～65535（Intel 系统），也就是我们所说的端口号。每一个寄存器分组视为一个寄存器数组，提供一个寄存器用于指定数组下标，再提供一个寄存器用于对索引所指向的数组元素（也就是寄存器）进行输入输出操作。这样用这两个寄存器就能够定位寄存器数组中的任何寄存器。
   
2. **VGA 寄存器访问模型**  
   
   - **Address Register**（索引端口）+ **Data Register**（数据端口）的双端口设计。  
   
     这两个寄存器就是各组中的 Address Register 和 Data Register。Address Register 作为数组的索引（下标），Data Register 作为寄存器数组中该索引对应的寄存器，它相当于所对应的寄存器的窗口，往此窗口读写的数据都作用在索引所对应的寄存器上。
   
     ![image-20250811201944180](F:\OS\xuexi\truth\图片\image-20250811201944180.png)
   
   - CRT Controller 组端口由 `Miscellaneous Output Register.I/OAS` 位决定：  
     - `I/OAS=1` → 索引端口 `0x3D4`，数据端口 `0x3D5`（彩色文本模式默认值）。  
     
     - 下表为CRT索引值对应寄存器
     
       ![image-20250811204134855](F:\OS\xuexi\truth\图片\image-20250811204134855.png)
   
3. **光标位置寄存器**  
   - 索引 `0x0e`（高 8 位）、`0x0f`（低 8 位），保存在 CRT Controller 寄存器组里。  

### 6.3.2 实现单个字符打印（put_char）  
#### 函数原型  
```c
void put_char(uint8_t char_asci);
```
#### 执行流程（print.S → put_char）  
1. **保存现场**  
   
   - `pushad` 备份 8 个 32 位寄存器。  
   
2. **装载视频段**  
   - 将 GDT 第 3 个描述符（DPL=0，基址 0xB8000）的选择子写入 `gs`，以便后续 `gs:offset` 直接寻址显存。  
   
3. **获取光标坐标**  
   - 通过 `out 0x3D4, 0x0e/0x0f` → `in 0x3D5` 读出 16 位线性坐标 → `bx`。  
   
4. **分类处理字符**  
   | 字符     | 动作          | 实现逻辑                        |
   | -------- | ------------- | ------------------------------- |
   | 普通字符 | 写显存+光标+1 | `mov [gs:bx*2], cl` 然后 `bx++` |
   | `\r`     | 回到行首      | `bx -= (bx % 80)`               |
   | `\n`     | 换到下一行    | `bx += 80`                      |
   | `\b`     | 退格          | `bx--`，再写空格 `0x0720`       |
   
5. **滚屏逻辑**  
   - 若 `bx ≥ 2000`（满屏），则：  
     1. `rep movsd` 把第 1~24 行搬到 0~23 行；  
     2. 填充最后一行 80 个空格 `0x0720`；  
     3. `bx = 1920`（第 24 行行首）。  
   
6. **更新光标寄存器**  
   - 再次通过 `0x3D4/0x3D5` 端口把 `bx` 写回 CRT Controller。  
   
7. **恢复现场**  
   - `popad` → `ret`。  
   
     ![image-20250811204344683](F:\OS\xuexi\truth\图片\image-20250811204344683.png)

![image-20250811210047852](F:\OS\xuexi\truth\图片\image-20250811210047852.png)

（1）新的光标值超出了屏幕右下角最后一个字符的位置。

（2）最后一行中任意位置有回车或换行符。

（3）新坐标超出屏幕范围

这些意味着我们要实现滚屏操作

（1）将第 1～24 行的内容整块搬到第 0～23 行，也就是把第 0 行的数据覆盖。

（2）再将第 24 行，也就是最后一行的字符用空格覆盖，这样它看上去是一个新的空行。

（3）把光标移到第 24 行也就是最后一行行首。

这里采用的是两种方案的第二种，具体见书P270

#### 细节 & 坑  

- **特权级问题**：用户进程 CPL=3 时若 `gs` 指向 DPL=0 的段会触发异常 → 方案：用户态打印走系统调用，内核态才直接改 `gs`。 用户进程是需要用 iretd 返回指令上 CPU 运行的， CPU 在执行 iretd 指令时会做特权检查，它检查 DS、ES、FS、GS“数据”段寄存器的内容。在 32 位保护模式下它们中存储的都是选择子，要是有任何一个段寄存器所指向的段描述符的 DPL权限高于从 iretd 命令返回后的 CPL（这个返回后的 CPL 也就是新的 CPL，CPL 就是加载到 CS 寄存器中选择子的 RPL），CPU 就会将该段寄存器赋值为 0。解决方法是：

  （1）为用户进程创建一个显存段描述符，其 DPL 为 3，专门给特权 3 级的用户进程使用。

  （2）在打印函数中动动手脚，将 GS 的值改为指向目前 DPL 为 0 的显存段描述符。

- **链接顺序**：`main.o` 在前，`print.o` 在后，否则符号解析会错位。  属于“调用在前，实现在后”的顺序。

  链接器最先处理的目标文件是参数中从左边数第一个（咱们这里是 main.o），对于里面找不到的符号（这里是 put_char），链接器会将它记录下来，以备在后面的目标文件中查找。如果将其顺序颠倒，势必导致在后面的目标文件，中才出现找不到符号的情况，而该符号的实现所在的目标文件早已经过去了，这可能使链接器内部采用另外的处理流程，导致出来的可执行程序不太一样。

### 6.3.4 实现整数打印（put_int）  

#### 函数原型  
```c
void put_int(uint32_t num);
```
#### 算法  
1. **进制**：固定十六进制，一次处理 4 bit → 1 个十六进制字符。  
2. **缓冲区**  
   - 8 B 的 `put_int_buffer` 存 8 个十六进制字符（大端字符序）。  
3. **转换**  
   - 掩码 `0x0F` 取最低 4 bit → 若 ≤9 加 `'0'`，否则减 10 加 `'A'`。  
4. **去前导 0**  
   - 从高位开始跳过 `'0'`，全 0 时保留一个 `'0'`。  
5. **输出**  
   - 逆序把有效字符逐个 `put_char`。  

###  速查  
| 函数       | 输入            | 依赖                               | 关键实现               | 输出效果         |
| ---------- | --------------- | ---------------------------------- | ---------------------- | ---------------- |
| `put_char` | 1 字节 ASCII    | `gs:0xB8000` 显存 + CRT 光标寄存器 | 写显存、更新光标、滚屏 | 单个字符显示     |
| `put_str`  | 0 结尾字符串    | 逐字节调 `put_char`                | 循环                   | 字符串显示       |
| `put_int`  | 32 位无符号整数 | 8 B 转换缓冲区 + `put_char`        | 十六进制转换+去前导0   | 无前缀十六进制数 |

到此，内核已拥有**完全自主的字符/字符串/整数**打印能力，为后续 `printf` 格式化输出奠定基础。

如果是 0～9 之间的数字，用该数字加上字符'0'的 ASCII 码 48。

如果是 A～F 之间的数字，用该数字减去 10 后再加上字符'A'的 ASCII 码 65。

- **大端字节序（Big-endian）**：**高位字节在前，低位字节在后**。也就是说，**最高有效字节（MSB）存放在最低的内存地址**。

  优点：符合人类阅读习惯（从左到右），网络协议（如TCP/IP）采用大端。

  缺点：某些处理器（如x86）处理效率较低。

- **小端字节序（Little-endian）**：**低位字节在前，高位字节在后**。也就是说，**最低有效字节（LSB）存放在最低的内存地址**。

  优点：现代处理器（如x86、ARM默认小端）处理效率更高，因为低位字节先处理。

  缺点：不符合人类阅读习惯，网络传输时需要转换（字节序转换函数如 `htons`, `htonl`）。

## 6.4 内联汇编

### 6.4.1 什么是内联汇编

内联汇编称为 inline assembly，GCC 支持在 C 代码中直接嵌入汇编代码，所以称为 GCC inline assembly。由于C 语言不支持寄存器操作，汇编语言可以，所以自然就想到了在 C 语言中嵌入内联汇编提升“战斗力”的方式，通过内联汇编，C 程序员可以实现 C 语言无法表达的功能，这样使开发能力大为提升。

| 种类         | 说明                                                  | 出现场景                     |
| ------------ | ----------------------------------------------------- | ---------------------------- |
| 基本内联汇编 | `asm("汇编指令")`，**无输入输出约束**                 | 简单指令、无需与 C 变量交互  |
| 扩展内联汇编 | `asm("汇编" : 输出 : 输入 : clobber)`，**有完整约束** | 需要参数、返回值、寄存器管理 |

### 6.4.2 AT&T 语法速览（Intel vs AT&T）
| 项         | Intel     | AT&T                          |
| ---------- | --------- | ----------------------------- |
| 寄存器     | `eax`     | `%eax`                        |
| 立即数     | `5`       | `$5`                          |
| 内存       | `[ebx+4]` | `4(%ebx)`                     |
| 操作数顺序 | 目的, 源  | 源, 目的                      |
| 大小后缀   | 无        | `b/w/l/q` → 字节/字/双字/四字 |

![image-20250813224815653](F:\OS\xuexi\truth\图片\image-20250813224815653.png)

寻址方式：内存寻址有固定的格式。

**segreg（段基址）：base_address(offset_address,index,size)** 

该格式对应的表达式为：

**segreg（段基址）：base_address+ offset_address+ index*size。**

base_address 是基地址，可以为整数、变量名，可正可负。

offset_address 是偏移地址，index 是索引值，这两个必须是那 8 个通用寄存器之一。

size 是个长度，只能是 1、2、4、8（Intel 语法中也是只能乘以这 4 个数）。

### 6.4.3 基本内联汇编

语法  
```c
asm [volatile] ("instruction1; instruction2;");
asm [volatile] ("assembly code")
```
- **volatile** 告诉 GCC 不要优化掉这段汇编。  

- “assembly code”是咱们所写的汇编代码，它必须位于圆括号中，而且必须用双引号引起来。

  （1）指令必须用双引号引起来，无论双引号中是一条指令或多条指令。

  （2）一对双引号不能跨行，如果跨行需要在结尾用反斜杠'\'转义。

  （3）指令之间用分号'；'、换行符'\n'或换行符加制表符'\n''\t'分隔。

- **只能操作全局变量**，局部变量在汇编层面不可见。  
  示例：直接调用 `write` 系统调用打印字符串（见 inlineASM.c）。

### 6.4.4 扩展内联汇编

基本内联汇编功能太弱，当汇编代码嵌入到 C 代码中，如果汇编代码想把 C 代码中的变量作为操作数加载到寄存器，如何找

到可用的寄存器，人为保存耗时耗力并且容易漏洞，故按照编译器格式进行 ，即约束。

完整模板  
```c
asm volatile ("汇编模板"
              : 输出操作数列表
              : 输入操作数列表
              : clobber/modify);

asm [volatile] (“assembly code”:output : input : clobber/modify)
```

output:output 用来指定汇编代码的数据如何输出给 C 代码使用。内嵌的汇编指令运行结束后，如果想将运行结果存储到 c 变量中，就用此项指定输出的位置。output 中每个操作数的格式为：“操作数修饰符约束名”（C 变量名）

input：input 用来指定 C 中数据如何输入给汇编使用。要想让汇编使用 C 中的变量作为参数，就要在此指定。input 中每个操作数的格式为：“[操作数修饰符] 约束名”（C 变量名）

#### 1. 操作数约束（constraint）

| **约束类别**   | **约束符** | **含义说明**                                                 |
| -------------- | ---------- | ------------------------------------------------------------ |
| **寄存器约束** | `a`        | 使用寄存器 `eax` / `ax` / `al`                               |
|                | `b`        | 使用寄存器 `ebx` / `bx` / `bl`                               |
|                | `c`        | 使用寄存器 `ecx` / `cx` / `cl`                               |
|                | `d`        | 使用寄存器 `edx` / `dx` / `dl`                               |
|                | `D`        | 使用寄存器 `edi` / `di`                                      |
|                | `S`        | 使用寄存器 `esi` / `si`                                      |
|                | `q`        | 任意通用寄存器之一（`eax`, `ebx`, `ecx`, `edx`）             |
|                | `r`        | 任意通用寄存器之一（`eax`, `ebx`, `ecx`, `edx`, `esi`, `edi`） |
|                | `g`        | 任意位置（寄存器或内存）                                     |
|                | `A`        | 使用 `eax:edx` 组合成64位整数                                |
|                | `f`        | 浮点寄存器                                                   |
|                | `t`        | 第1个浮点寄存器（`st(0)`）                                   |
|                | `u`        | 第2个浮点寄存器（`st(1)`）                                   |
| **内存约束**   | `m`        | 任意内存地址（作为指针使用）                                 |
|                | `o`        | 内存地址，且支持偏移量访问                                   |
| **立即数约束** | `i`        | 任意整数立即数                                               |
|                | `F`        | 浮点立即数                                                   |
|                | `I`        | 0~31 之间的立即数                                            |
|                | `J`        | 0~63 之间的立即数                                            |
|                | `N`        | 0~255 之间的立即数                                           |
|                | `O`        | 0~32 之间的立即数                                            |
|                | `X`        | 任意类型的立即数                                             |
| **通用约束**   | `0`~`9`    | 与第 n 个操作数使用相同的寄存器或内存位置（仅用于 input）    |

包括寄存器约束，内存约束，立即数约束，通用约束。

#### 2. 占位符

- **序号占位符**：`%0 %1 … %9` 对应操作数出现顺序。  
- 扩展内联汇编会在前面加两个%用以区分占位符和寄存器。
- **名称占位符**：`[name]` + `%[name]` 引用，易读。  
- **寄存器子模式**：  
  - `%b0` → 低 8 位（`al`）  
  - `%h0` → 高 8 位（`ah`）  
  - `%w0` → 低 16 位（`ax`）  
  - `%k0` → 32 位（`eax`）

#### 3. 输出/输入修饰符
| 修饰符 | 作用             | 出现位置 |
| ------ | ---------------- | -------- |
| `=`    | 只写             | 输出     |
| `+`    | 读写             | 输出     |
| `&`    | 独占寄存器       | 输出     |
| `%`    | 可与下一输入交换 | 输入     |

#### 4. clobber / modify
- **寄存器**：`"eax"`, `"ebx"` …  
- **标志位**：`"cc"`  
- **内存**：`"memory"`（告诉 GCC 内存被改，需重读缓存）

### 6.4.5 机器模式（Machine Mode）
用于 **精确指定数据宽度/寄存器子部分**。  
| 后缀 | 对应宽度 | 例子   |
| ---- | -------- | ------ |
| `b`  | 1 B      | `%al`  |
| `w`  | 2 B      | `%ax`  |
| `k`  | 4 B      | `%eax` |
| `q`  | 8 B      | `%rax` |

使用场景：  
```c
asm("movw %w1, %0" : "=m"(dst) : "r"(src));
```
`%w1` 强制把 `src` 的低 16 位搬过去，即使 `src` 是 32 位寄存器。



| 场景                              | 推荐写法                          |
| --------------------------------- | --------------------------------- |
| 只想插入几条指令，不与 C 变量交互 | 基本内联汇编 `asm("nop");`        |
| 需要 C 变量 ↔ 汇编寄存器/内存     | 扩展内联汇编 + 约束               |
| 不确定寄存器                      | 用 `"r"` 让 GCC 分配，占位符 `%0` |
| 读写同一变量                      | `"+"` 修饰符，避免两次约束        |
| 汇编里调用函数后需保护返回值      | `=&a(result)` 防寄存器复用        |
| 修改内存或标志位                  | clobber 里加 `"memory"` 或 `"cc"` |
