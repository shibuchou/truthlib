# 第四章：保护模式入门

### 4.1 保护模式概述  

**为什么要有保护模式？**  

实模式只有 1 MB 空间、无权限隔离，一次只能跑一个任务。  

实模式的缺点：

（1）实模式下操作系统和用户程序属于同一特权级，这哥俩平起平坐，没有区别对待。

（2）用户程序所引用的地址都是指向真实的物理地址，也就是说逻辑地址等于物理地址，实实在在地指哪打哪。

（3）用户程序可以自由修改段基址，可以不亦乐乎地访问所有内存，没人拦得住。

以上 3 个原因属于安全缺陷，没有安全可言的 CPU 注定是不可依赖的，这从基因上决定了用户程序乃至操作系统的数据都可以被随意地删改，一旦出事往往都是灾难性的，而且不容易排查。

（4）访问超过 64KB 的内存区域时要切换段基址，转来转去容易晕乎。

（5）一次只能运行一个程序，无法充分利用计算机资源。

（6）共 20 条地址线，最大可用内存为 1MB，这即使在 20 年前也不够用。

保护模式提供：  
 32 位寻址（4 GB）  
 特权级隔离（0-3 级）  
 段/页双重保护  
多任务、异常、分页、虚拟内存  

**“实模式不是 32 位 CPU”** 的梗：CPU 上电后默认 16 位，需要软件主动打开 PE 位才能进入 32 位世界。

### 4.2 初见保护模式（4 个“扩展”）  
| 扩展类别         | 实模式     | 保护模式变化                                           |
| ---------------- | ---------- | ------------------------------------------------------ |
| **寄存器扩展**   | 16 位      | 变成 32 位（EAX/EBX…），低 16 位保持兼容               |
| **寻址扩展**     | 段×16+偏移 | 段寄存器里放“选择子”，通过描述符得 32 位段基址         |

| **指令扩展**     | 16 位指令  | 新增 `mov eax, [eax+ecx*4]` 这类 32 位寻址指令         |

### 4.3 全局描述符表 GDT（打开保护模式的钥匙）  

1. **段描述符**（8 字节结构）  

   段基址 32 位、段限长 20 位、属性 12 位  

   属性字段：G（粒度）、D/B（缺省操作大小）、P（存在位）、DPL（特权级）、S（代码/数据段）、Type（可读/可写/可执行…）  

   ![image-20250717221245947](F:\OS\xuexi\truth\图片\image-20250717221245947.png)

   实际的段界限边界值  = （描述符中段界限+1）*（段界限的粒度大小：4KB 或者 1）-1。

   如果 G 位为 0，表示段界限粒度大小为 1 字节，如果 G 位为 1，表示段界限粒度大小为 4KB 字节。

   先在cs中找到选择子，通过选择子作为偏移在gdt表中查找到对应的段描述符（GDT表中的**每一个条目**（8字节），描述了一个内存段的详细信息），从而找到段基址，再结合IP中的偏移，CPU将逻辑地址（选择子+偏移）转换为线性地址（段基址+偏移）。

2. **GDT/LDT/选择子**  

   **LDT 是每个任务（进程）私有的“段描述符表”，用于定义该任务自己的代码段、数据段等，其他任务看不到也访问不到，与 GDT（全局描述符表）功能类似，但作用范围更小、更灵活。**

   | 项目         | GDT（全局描述符表）               | LDT（局部描述符表）                     |
   | ------------ | --------------------------------- | --------------------------------------- |
   | **作用域**   | 系统全局，所有任务共享            | 每个任务私有，互不干扰                  |
   | **数量**     | 整个系统只有一个                  | 每个任务可以有一个（也可以没有）        |
   | **存放内容** | 系统段、共享段、TSS、LDT 描述符等 | 任务私有的代码段、数据段、栈段等        |
   | **寄存器**   | 由 `GDTR` 指向                    | 由 `LDTR` 指向                          |
   | **地址来源** | 直接由 GDTR 提供基址              | LDT 本身是一个段，其描述符存放在 GDT 中 |
   | **是否必须** | 必须使用                          | 可选使用（现代 OS 通常不用）            |

   GDT 存于内存，GDTR 寄存器存“基址+限长”。  进入保护模式需要有 GDT，但进入保护模式后，还可以再重新换个 GDT 加载。

   | 寄存器   | 位数                              | 内容/结构                                     | 指向的目标            | 是否直接给出内存地址   | 典型用途                               |
   | -------- | --------------------------------- | --------------------------------------------- | --------------------- | ---------------------- | -------------------------------------- |
   | **GDTR** | 48 位                             | 32 位线性基址 + 16 位界限                     | **GDT 本身**          | ✅ 直接给出             | 系统初始化时由 OS 一次性设置           |
   | **LDTR** | 16 位（可见） + 64 位（隐藏缓存） | 一个 **段选择子**（指向 GDT 中的 LDT 描述符） | **当前任务的 LDT 段** | ❌ 间接给出（通过 GDT） | 每次任务切换时由 OS 用 `lldt` 指令更新 |

   系统启动时执行 `lgdt [gdt_desc]` 把 GDT 的基址和界限装入 GDTR。以后所有 TI=0 的选择子都直接根据 GDTR 去 **GDT** 里查描述符。

   使用 LDTR（LDT）操作系统为每个任务建立一张 LDT，并在 GDT 中放一项“LDT 描述符”。任务切换时，OS 执行 `lldt selector_to_ldt_desc`→ 该 selector 写入 LDTR（16 位可见部分）。→ CPU 自动把对应的 LDT 描述符（基址/界限/属性）缓存到 LDTR 的影子寄存器。以后所有 TI=1 的选择子都走“LDTR → GDT → LDT”这条二级查找路径。

   LDTR 里只是一个 16 位的**选择子**（index+TI+RPL）CPU 把它的 index 拿去 GDT 里查，得到一条**LDT 段描述符**；这条描述符给出**整个 LDT 表**的**32 位基址**和**20 位限长**。**再用“真正的段选择子（存在cs中）” → LDT**之后，当某条指令用到的段选择子的 TI 位 = 1 时，CPU 才用刚才找到的 LDT 基址，加上这个新选择子的 index×8，到 LDT 里去取**目标段描述符**；最后把描述符里的 **段基址 + 程序员给出的偏移(offset)** 拼成线性地址。

   CPU 根据“段寄存器里的选择子”索引 GDT，得到段基址+属性。  

   **选择子格式**

   TI=0 查 GDT，TI=1 查 LDT；RPL 表示请求特权级0,1,2,3四种。

3. **打开保护模式三步骤**  

   1. 构建 GDT 并 `lgdt` 加载到 GDTR。  

   2. **打开 A20 地址线**（让第 21 位有效，避免回卷）。  将端口 0x92 的第 1 位置 1 就可以：

      ```
      in al，0x92 
      
      or al，0000_0010B 
      
      out 0x92，al 
      ```

      

   3. **置 CR0 的 PE 位 = 1** → CPU 进入保护模式。  

      ![image-20250717233353178](F:\OS\xuexi\truth\图片\image-20250717233353178.png)

      ![image-20250717233407585](F:\OS\xuexi\truth\图片\image-20250717233407585.png)

      ```
       mov eax, cr0 ;将 cr0 写入 eax。
      
       or eax, 0x00000001 ;通过或运算 or 指令将 eax 的第 0 位置 1。
      
       mov cr0, eax ;将 eax 写回 cr0，这样 cr0 的 PE 位便为 1 了。
      ```

      

   4. **远跳转**刷新流水线，加载新的段描述符缓存。

### 4.4 处理器微架构简介（流水线 & 缓存）  
| 机制            | 作用             | 对 OS 的意义              |
| --------------- | ---------------- | ------------------------- |
| **流水线**      | 多条指令重叠执行 | 远跳转/段切换会清空流水线 |
| **乱序执行**    | 提高并行度       | 需保证内存一致性          |
| **L1/L2 Cache** | 缓物理内存       | 维护一致性的 flush 指令   |
| **分支预测**    | 减少跳转损失     | 远跳转可强制清空预测器    |

### 4.5 用远跳转清空流水线  
进入保护模式后执行 `jmp 0x08:PM_entry` 之类 **远跳转**，作用：  
1. 刷新指令预取队列  
2. 更新段描述符缓存寄存器（CS 隐藏部分）  
3. 确保后续指令使用 32 位译码路径

### 4.6 内存段保护机制  
#### 4.6.1 向段寄存器加载选择子时的检查  
- **索引越界** → #GP(0)  **General-Protection Fault（一般保护错误）**，括号里的 **0** 表示错误码为 0，即 **没有提供段选择子** 或 **选择子无效**。

- **描述符类型/特权级不符** → #GP(selector)

  ![image-20250718001210885](F:\OS\xuexi\truth\图片\image-20250718001210885.png)

#### 4.6.2 代码段 & 数据段的保护  
| 检查项                 | 触发异常      |
| ---------------------- | ------------- |
| 写只读代码段           | #GP(0)        |
| 执行数据段             | #GP(0)        |
| CPL > DPL 或 RPL > DPL | #GP(selector) |

#### 4.6.3 栈段的特殊规则  
- 栈段描述符必须 Type = 可读/写数据段  
- 向 **向下扩展** 的栈段写入时，偏移必须 **> limit**，否则 #SS   **Stack Segment Fault（栈段错误）**，向量号 **12**。它专门在 **涉及栈段（SS）的操作出错** 时触发
- `push`/`pop` 硬件自动检查限长 → 触发 #SS 或 #GP

### 总结  
> 第四章把 CPU 从 16 位“裸奔”实模式，**升级成 32 位“带保镖”的保护模式**：  
> **建立 GDT → 打开 A20 → 置 PE → 远跳转** 四步曲，让你拥有 4 GB 空间、特权级保护和硬件段检查，为后面的分页、内核、任务切换铺平道路。







**段描述符设置固定还是可变，mbr具体加载流程，第0章**

###  **段描述符表的位置不是固定的**
- **GDT（全局描述符表）**和**LDT（局部描述符表）**是内存中的一段数据结构，**可以放在内存的任意位置**，只要满足对齐要求（GDT需8字节对齐）。
- 操作系统或引导程序负责：
  - 在内存中**申请一块空间**作为GDT；
  - **构造段描述符**（如代码段、数据段、栈段等）；
  - 用 `lgdt` 指令把GDT的**起始地址和界限**加载到GDTR寄存器。

###  **段描述符的内容也不是固定的**
每个段描述符（8字节）包含：
- 段基址（Base）

- 段界限（Limit）

- 访问权限（DPL、类型、粒度等）

  这些字段**由操作系统根据需要动态设置**，例如：

- 内核代码段：基址=0，界限=4GB，权限=可读可执行
- 用户数据段：基址=0，界限=4GB，权限=可读可写

**GDTR寄存器**（48位）结构是固定的：

- 低16位：GDT的界限（大小-1）
- 高32位：GDT的线性基址

但**这个寄存器里的值是运行时写入的**，不是硬件写死。



### **mbr具体加载流程**：

1. BIOS 把磁盘 0 面 0 道 1 扇区（LBA 0）512 B 读到 0x7C00 并跳过去。
   · 这一段就是 MBR。
2. MBR 运行后只做 3 件事：
   ① 把自己所有段寄存器设为 0x0000，栈顶放到 0x7C00（或更低）
   ② 用 BIOS INT 13h 把“下一个”扇区（LBA 1，即 0 面 0 道 2 扇区）读进内存
   常用地址：0x8000、0x9000、0x7E00 等（避开 0x7C00–0x7DFF 的 MBR 本身）。
   ③ 读完最后一个 Loader 扇区后，`jmp 0x0000:LOADER_OFFSET` 把控制权交给 Loader。
3. Loader 从此开始跑，后续再加载真正的内核。



### 总结:

第四章的核心任务是将CPU从实模式切换到保护模式，解决实模式下内存空间小（仅1MB）、无权限隔离、无法多任务等根本缺陷。首先系统介绍了保护模式的新机制：32位寄存器扩展、灵活的段页式寻址、特权级隔离（0-3级）、以及段描述符和全局描述符表（GDT）等关键数据结构。然后通过实例，完成从实模式到保护模式的完整切换流程：构建并加载GDT、打开A20地址线、置CR0寄存器的PE位，最后通过远跳转刷新流水线并初始化32位环境。为后续分页、内核和多任务机制打下基础，也首次体验到保护模式带来的4GB地址空间和硬件级内存保护，也是正式进入了我们对内核的下一步探索和部署。
